## Overview

Skygen uses a token-based billing model. Tokens are consumed when the agent makes LLM API calls and when Cloud PC sessions are running. Each subscription plan includes a monthly token allocation. Free users receive a periodic renewable token grant. Paid users can enable overage billing to continue working after their included tokens run out.

---

## Subscription tiers

| Tier | Description |
|------|-------------|
| `free` | Freemium plan. Token grant renews periodically. No card required. |
| `standard` | Paid subscription with monthly token allocation and optional overage. |
| `none` | No subscription. Access blocked. |

The `trial` tier is legacy — all trial users were migrated to `free` as of migration `20260318_remove_trial`.

---

## Plan structure

Plans are configured via the admin panel and stored in `subscription_plans`. Each plan has:

| Field | Description |
|-------|-------------|
| `name` | Display name (e.g. `Lite`, `Pro`, `Enterprise`) |
| `price_usd` | Monthly price in USD |
| `monthly_tokens` | Tokens included per billing period |
| `billing_period_days` | Billing period length (default: 30 days) |
| `allow_overage` | Whether the plan supports overage billing |
| `model_profile` | Agent model profile: `fast`, `default`, or `pro` |
| `allowed_agent_modes` | JSON array of allowed modes (e.g. `["auto", "cua", "llm"]`) |
| `features_config` | JSON object with feature flags (e.g. `{"cloud_pc": true, "max_cloud_pcs": 3}`) |
| `daily_usd_limit` | Free plan daily spending cap in USD |
| `limit_renewal_period_days` | Free plan renewal period |

---

## Free tier mechanics

Free users get a renewable token grant defined by the plan's `daily_usd_limit` and `limit_renewal_period_days`. The system:

1. Checks the user's current token balance.
2. If below the minimum threshold (2 tokens), looks up when the last `free_grant` transaction was created.
3. If the renewal period has elapsed, grants a new batch of tokens (transaction type: `FREE_GRANT`).
4. If the period has not elapsed, returns `free_limit_reached` with the next renewal timestamp.

```python
# AccessResult returned when free tier is exhausted
{
  "can_access": False,
  "reason": "free_limit_reached",
  "extra": {
    "next_renewal": "2026-05-09T10:00:00Z",
    "plan_name": "Free"
  }
}
```

## Token consumption

Tokens are deducted for two categories of usage:

### LLM API usage

Each LLM call from the agent costs tokens proportional to the API cost in USD. The exact conversion rate is set server-side. Transaction type: `API_USAGE`.

### Cloud PC usage

Cloud PC sessions consume tokens per minute while running. When a session starts, a recurring per-minute deduction is recorded. Transaction type: `CLOUD_PC_USAGE`.

---

## Token balance

Every user has a single `TokenBalance` row with:

| Field | Description |
|-------|-------------|
| `balance` | Current integer token count |
| `fractional_tokens` | Accumulated sub-token fractional amounts not yet deducted |
| `total_earned` | Lifetime tokens credited |
| `total_spent` | Lifetime tokens consumed |

---

## Token transactions

All token movements are recorded in `TokenTransaction`:

| Type | Direction | Description |
|------|-----------|-------------|
| `subscription_topup` | Credit | Monthly token grant on subscription renewal |
| `admin_grant` | Credit | Manual grant by admin |
| `promo_code` | Credit | Promotional tokens |
| `refund` | Credit | Refunded tokens |
| `free_grant` | Credit | Periodic free plan token grant |
| `api_usage` | Debit | LLM API calls |
| `cloud_pc_usage` | Debit | Cloud PC per-minute charge |
| `manual_deduction` | Debit | Admin deduction |

## Overage billing

Paid subscribers can opt into overage billing to continue using the service after their included tokens run out. Overage is billed as pay-as-you-go via Stripe metered billing:

- `allow_overage = True` must be set on the subscription.
- Overage tokens used are tracked in `overage_tokens_used`.
- A Stripe metered billing item (`stripe_metered_item_id`) records usage and charges monthly.

Users enable overage on the Pricing page. The `STANDARD` tier supports overage; the `FREE` tier does not.

## Subscription management

### View current status

```http
GET /billing/status
Authorization: Bearer {token}
```

Returns the user's active subscription, token balance, and Stripe payment method status.

### List available plans

```http
GET /billing/plans
Authorization: Bearer {token}
```

### Subscribe to a plan

```http
POST /billing/subscribe
Authorization: Bearer {token}

{"plan_id": "plan-uuid"}
```

For paid plans, this creates a Stripe Checkout session. The user is redirected to Stripe to complete payment.

### Cancel subscription

```http
POST /billing/cancel
Authorization: Bearer {token}
```

Sets `cancel_at_period_end = True`. The subscription remains active until the period ends.

### View token transaction history

```http
GET /billing/transactions
Authorization: Bearer {token}
```

## Model profiles

Plans assign a `model_profile` that controls which LLM the agent uses:

| Profile | Typical models | Use case |
|---------|---------------|---------|
| `fast` | Gemini Flash, Qwen Fast | Quick tasks, low cost |
| `default` | Claude Sonnet, Gemini Pro | Balanced performance |
| `pro` | Claude Opus, GPT-4o | Complex reasoning tasks |

Users on plans with `allowed_model_profiles` set can choose their profile in the settings. The public display name (e.g., "Skygen Fast", "Skygen Pro") is configured via the admin panel's `ModelProfileConfig` table.

## Stripe integration

Skygen uses Stripe for payment processing. Webhook events are received at `POST /billing/webhooks` and verified using `STRIPE_WEBHOOK_SECRET`. Supported events:

| Stripe event | Action |
|-------------|--------|
| `checkout.session.completed` | Activate subscription |
| `invoice.payment_succeeded` | Add monthly tokens, reset overage counter |
| `invoice.payment_failed` | Mark subscription `past_due` |
| `customer.subscription.deleted` | Cancel subscription |
| `customer.subscription.updated` | Sync plan changes |
