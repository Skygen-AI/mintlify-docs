## Overview

Triggers let you automate agents without manual intervention. A trigger watches for events from connected integrations (new Gmail message, new Slack mention, new GitHub issue) and fires an automation instruction to the agent when the event occurs. Triggers are backed by Composio's event pipeline with HMAC-verified webhook delivery.

---

## How triggers work

1. You create a trigger specifying a toolkit (e.g., `gmail`), an event type (e.g., `new_email`), and an `action_prompt` describing what the agent should do.
2. Skygen registers the trigger with Composio via `POST /mcp/triggers`.
3. When the event fires, Composio sends a signed webhook to `POST /mcp/webhooks`.
4. The backend verifies the HMAC signature, parses the event, and dispatches it to the agent via Socket.IO.
5. The agent runs the `action_prompt` in the context of the event payload.

## Creating a trigger

```http
POST /mcp/triggers
Authorization: Bearer {token}
Content-Type: application/json
```

```json
{
  "toolkit": "gmail",
  "trigger_name": "new_email",
  "config": {
    "filters": {"from": "boss@company.com"}
  },
  "action_prompt": "Read the new email and summarize it, then add it to my daily digest doc in Google Drive.",
  "account_id": "gmail-account-uuid"
}
```

<ParamField body="toolkit" type="string" required>
  Composio toolkit slug (e.g. `gmail`, `slack`, `github`).
</ParamField>

<ParamField body="trigger_name" type="string" required>
  Event type within the toolkit (e.g. `new_email`, `new_message`, `new_issue`).
</ParamField>

<ParamField body="config" type="object">
  Trigger-specific configuration passed to Composio. Shape depends on the toolkit and event type.
</ParamField>

<ParamField body="action_prompt" type="string">
  Natural language instruction for the agent when this trigger fires. The event payload is injected as context.
</ParamField>

<ParamField body="account_id" type="string">
  Optional. For toolkits with multiple connected accounts, specifies which account to watch.
</ParamField>

**Response — 201 Created:**

```json
{
  "trigger_id": "composio-trigger-uuid",
  "trigger_name": "new_email",
  "app": "gmail",
  "status": "active",
  "config": {"filters": {"from": "boss@company.com"}},
  "action_prompt": "Read the new email and summarize it...",
  "is_active": true,
  "created_at": "2026-05-08T10:00:00Z"
}
```

## Listing triggers

```http
GET /mcp/triggers
Authorization: Bearer {token}
```

Returns all triggers for the current user. If Composio is temporarily unavailable, returns an empty list with `composio_status: "unavailable"` (not an error response) so the UI can show a retry hint rather than treating it as "no triggers configured".

---

## Updating a trigger

```http
PATCH /mcp/triggers/{trigger_id}
Authorization: Bearer {token}

{
  "action_prompt": "Summarize the email and add a TODO to my Notion.",
  "is_active": true
}
```

<ParamField body="action_prompt" type="string">
  Updated automation instruction.
</ParamField>

<ParamField body="is_active" type="boolean">
  Enable or disable the trigger without deleting it.
</ParamField>

---

## Deleting a trigger

```http
DELETE /mcp/triggers/{trigger_id}
Authorization: Bearer {token}
```

Removes the trigger from Composio and Skygen's local record.

---

## Viewing trigger event history

```http
GET /mcp/triggers/{trigger_id}/events
Authorization: Bearer {token}
```

Returns the history of events that fired for this trigger, including status and error messages.

<ResponseField name="events[].id" type="string">
  Event record UUID.
</ResponseField>

<ResponseField name="events[].trigger_id" type="string">
  Parent trigger ID.
</ResponseField>

<ResponseField name="events[].event_type" type="string">
  Event type (e.g. `new_email`).
</ResponseField>

<ResponseField name="events[].status" type="string">
  `pending`, `dispatched`, `failed`.
</ResponseField>

<ResponseField name="events[].payload" type="object">
  Raw event data from Composio (email metadata, message content, etc.).
</ResponseField>

<ResponseField name="events[].error_message" type="string">
  Error detail if status is `failed`.
</ResponseField>

## Webhook verification

Incoming Composio webhooks are verified with HMAC-SHA256. The `COMPOSIO_WEBHOOK_SECRET` env var is required in production. Without it, webhooks are rejected in production and only accepted in development (with a warning).

Signature verification:

```python
expected = hmac.new(secret.encode(), payload_bytes, hashlib.sha256).hexdigest()
if not hmac.compare_digest(expected, received_signature):
    raise WebhookVerificationError("Invalid signature")
```

## Available trigger events

Common events by toolkit:

| Toolkit | Trigger name | Description |
|---------|-------------|-------------|
| `gmail` | `new_email` | New email received |
| `gmail` | `new_thread` | New email thread started |
| `slack` | `new_message` | Message posted to a channel |
| `slack` | `new_mention` | User mentioned in a message |
| `github` | `new_issue` | Issue opened in a repo |
| `github` | `new_pull_request` | PR opened in a repo |
| `github` | `push` | Code pushed to a branch |
| `linear` | `new_issue` | Issue created in Linear |
| `notion` | `page_updated` | A Notion page is updated |

The full list of available triggers for any toolkit can be fetched from Composio's API. Only toolkits you have connected produce events.

## Trigger action prompt context

When a trigger fires, the agent receives:
- The `action_prompt` you defined.
- The raw event `payload` from Composio (e.g., email subject, sender, body preview).
- Access to all your connected MCP toolkits for the automation.

Write prompts as natural instructions referencing the event:

```
When a new GitHub issue is opened:
"Label the issue by reading its body, then assign it to the appropriate team member based on the content."
```
