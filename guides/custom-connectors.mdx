## Overview

Custom connectors allow you to expose any HTTP API as a set of tools the Skygen agent can invoke. A connector has a type (OpenAPI, REST, or GraphQL), a name, a slug, authentication config, and a list of tool definitions. The agent discovers tools via the connector registry and calls them during task execution.

---

## Connector types

| Type | Description |
|------|-------------|
| `openapi` | Connector introspects an OpenAPI 3.x spec URL and auto-generates tool definitions |
| `rest` | Manual tool definitions for REST endpoints |
| `graphql` | Manual tool definitions wrapping GraphQL operations |

---

## Creating a connector

```http
POST /connectors
Authorization: Bearer {token}
Content-Type: application/json
```

```json
{
  "type": "openapi",
  "name": "My CRM API",
  "description": "Customer relationship management API",
  "config": {
    "base_url": "https://api.mycrm.com/v1",
    "spec_url": "https://api.mycrm.com/v1/openapi.json",
    "auth_type": "bearer",
    "auth_token": "sk-..."
  }
}
```

<ParamField body="type" type="string" required>
  Connector type: `openapi`, `rest`, or `graphql`.
</ParamField>

<ParamField body="name" type="string" required>
  Human-readable name. Max 100 characters.
</ParamField>

<ParamField body="description" type="string">
  Description shown to the agent in tool discovery.
</ParamField>

<ParamField body="config" type="object" required>
  Authentication and endpoint configuration. Encrypted at rest using AES (key: `MCP_CRYPTO_KEY`).
</ParamField>

<ParamField body="note" type="string">
  Private note visible only to the owner. Not shown to the agent.
</ParamField>

**Response — 201 Created:**

```json
{
  "id": "connector-uuid",
  "user_id": "user-uuid",
  "type": "openapi",
  "name": "My CRM API",
  "slug": "my-crm-api",
  "description": "Customer relationship management API",
  "icon_url": null,
  "config": {"base_url": "...", "spec_url": "...", "auth_type": "bearer", "auth_token": "[REDACTED]"},
  "tools": null,
  "status": "active",
  "created_at": "2026-05-08T10:00:00",
  "updated_at": "2026-05-08T10:00:00"
}
```

<Note>
  The `config` field in responses always has secrets replaced with `[REDACTED]`. The plaintext is only used when the agent invokes a tool.
</Note>

## Config authentication options

| `auth_type` | Required fields | Notes |
|-------------|----------------|-------|
| `bearer` | `auth_token` | `Authorization: Bearer {auth_token}` |
| `api_key` | `api_key`, `api_key_header` | Custom header name |
| `basic` | `username`, `password` | Base64-encoded |
| `none` | — | Public API |

## Listing connectors

```http
GET /connectors
Authorization: Bearer {token}
```

Returns all connectors owned by the authenticated user. Sensitive config values are redacted.

---

## Updating a connector

```http
PUT /connectors/{connector_id}
Authorization: Bearer {token}
Content-Type: application/json
```

All fields are optional. Only provided fields are updated.

---

## Deleting a connector

```http
DELETE /connectors/{connector_id}
Authorization: Bearer {token}
```

Removes the connector. Running tasks that reference this connector may fail.

## Uploading a connector icon

Connector icons are stored in S3. Use the two-step presigned upload flow:

**Step 1: Get presigned URL**

```http
POST /connectors/icon/presigned
Authorization: Bearer {token}

{"content_type": "image/png", "file_extension": "png"}
```

Returns `upload_url`, `fields`, and `icon_key`.

**Step 2: Upload to S3**

```python
import requests

resp = requests.post(
    presigned["upload_url"],
    data=presigned["fields"],
    files={"file": open("icon.png", "rb")},
)
```

**Step 3: Set icon on connector**

```http
PATCH /connectors/{connector_id}/icon
Authorization: Bearer {token}

{"icon_key": "connector-icons/{user_id}/{uuid}.png"}
```

Supported icon types: PNG, JPEG, SVG, WebP. Maximum size: 2 MB.

## Manual tool definitions

For `rest` type connectors, define tools explicitly:

```json
{
  "type": "rest",
  "name": "Internal API",
  "config": {
    "base_url": "https://internal.example.com/api",
    "auth_type": "bearer",
    "auth_token": "internal-token"
  },
  "tools": [
    {
      "name": "get_customer",
      "description": "Retrieve a customer by ID",
      "method": "GET",
      "path": "/customers/{customer_id}",
      "parameters": {
        "customer_id": {"type": "string", "description": "Customer UUID", "required": true}
      }
    },
    {
      "name": "create_ticket",
      "description": "Create a support ticket",
      "method": "POST",
      "path": "/tickets",
      "body_schema": {
        "title": {"type": "string", "required": true},
        "priority": {"type": "string", "enum": ["low", "normal", "high"]}
      }
    }
  ]
}
```

## Security

All connector configs are encrypted at rest. The encryption key is `MCP_CRYPTO_KEY` (AES). The `connectors_crypto.py` service handles encrypt/decrypt/redact operations transparently — application code never sees plaintext secrets outside of active tool invocation.

SSRF protection: the `validate_url_not_ssrf()` service validates all base URLs and spec URLs against a blocklist before saving. Requests to localhost, link-local addresses (169.254.0.0/16), and internal RFC 1918 ranges are rejected.

## Connector status values

| Status | Meaning |
|--------|---------|
| `active` | Connector is live and available to agents |
| `inactive` | Connector is disabled; not offered to agents |
| `error` | Last introspection or tool call failed |
