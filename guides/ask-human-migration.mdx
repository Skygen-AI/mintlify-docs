## Overview

Skygen has replaced the original HTTP-poll + Socket.IO `confirmation_request` event system with the hibernate/wake ask-human pattern (ARCH-001). The new system is more reliable: the agent pod sleeps in Redis while waiting, uses atomic Redis claims to prevent duplicate wakes across pods, and derives `waiting_for_input` from database rows rather than managing it imperatively.

This guide explains what changed and how to update any code that relied on the old confirmation flow.

---

## What changed

### Before (legacy)

The old confirmation system worked as follows:

1. Agent called a backend endpoint to register a confirmation request.
2. Backend stored the request and emitted `confirmation_request` via Socket.IO to the user's namespace.
3. The agent polled `GET /safety/confirmation/{violation_id}` in a loop until the user responded.
4. The user responded via the UI, which called `POST /safety/confirmation/{violation_id}/respond`.
5. The backend's `SafetyService` managed the `waiting_for_input` flag imperatively.

Problems with this approach:
- The agent pod consumed CPU during the polling wait.
- Multiple pods could race on the same confirmation, delivering the response multiple times.
- Adding a new resolution path (e.g., OAuth callback) required re-implementing `waiting_for_input` flag management.
- Flag could drift out of sync when new resolution paths missed the manual flip.

### After (current — ARCH-001)

1. Agent calls `communicate(wait_for_reply=True)` which emits a `ui_meta(ask_human)` payload.
2. The backend's `BrokerAgent` calls `HibernationController.Hibernate()` to write a `SentinelTask` to Redis.
3. The agent worker loop exits. The pod thread is freed.
4. `ask_human_state.mark_ask_human_pending(session_id)` derives `waiting_for_input = True` from database rows.
5. When the user replies (or OAuth completes), `WakeSession()` / `WakeUserSessions()` atomically claims and fires the wake.
6. `mark_ask_human_resolved()` flips the relevant `ChatMessage` rows to `answered` and recomputes the flag.

## Migration steps

### 1. Remove calls to legacy safety confirmation endpoints

The following endpoints have been deprecated and should no longer be called from agent code:

```
# DEPRECATED — do not use
POST /safety/confirmation
GET  /safety/confirmation/{violation_id}
POST /safety/confirmation/{violation_id}/respond
```

These were in `app/services/safety_service.py` under the historical note:
> "Human-confirmation flows used to live here via an HTTP-poll + Socket.IO `confirmation_request` event."

### 2. Replace agent polling with communicate()

Old pattern in agent code:

```python
# LEGACY — do not use
violation_id = await self._register_confirmation(question)
while True:
    response = await self._poll_confirmation(violation_id)
    if response:
        break
    await asyncio.sleep(2)
```

New pattern:

```python
# CURRENT — use this
result = await self.communicate(
    message=question,
    wait_for_reply=True,
    ui_meta={
        "type": "ask_human",
        "input": {"type": "input", "placeholder": "Type your answer..."},
    },
)
# Agent hibernates here; execution resumes after wake with result containing user reply
```

The `communicate()` call emits the `ui_meta` payload to the frontend and then calls `HibernationController.Hibernate()`. The agent worker does not continue until the `WakeSession()` event arrives.

### 3. Replace manual waiting_for_input flag management

Old pattern:

```python
# LEGACY — imperative flag management
session.meta_data["waiting_for_input"] = True
await db.commit()
await emit_session_updated(session)
# ... later ...
session.meta_data["waiting_for_input"] = False
await db.commit()
```

New pattern — do nothing. The flag is derived automatically:

```python
# CURRENT — call these helpers; flag management is automatic
await mark_ask_human_pending(session_id)      # after inserting ask_human message
await mark_ask_human_resolved(session_id)     # after user replies
```

Both helpers call `_recompute_and_emit()` which queries the database for pending `ask_human_action = "request"` rows and emits `session_updated` only when the flag actually changes.

### 4. Update confirmation_request Socket.IO listeners

If your frontend subscribed to `confirmation_request` events:

```javascript
// LEGACY
socket.on("confirmation_request", (data) => {
  showConfirmationModal(data);
});
```

Replace with the `session_updated` + message metadata pattern:

```javascript
// CURRENT
socket.on("session_updated", (session) => {
  if (session.meta_data?.waiting_for_input) {
    // Check latest ask_human message in the session for the prompt
  }
});

socket.on("chat_message_meta_updated", ({ session_id, message_id, meta_data }) => {
  if (meta_data.ask_human_action === "answered") {
    // Clear ask_human UI for this message
  }
});
```

### 5. Update resolution paths

Any code path that resolves a user confirmation must now call the appropriate resolver:

| Situation | Old call | New call |
|-----------|----------|---------|
| User typed a reply | Manual flag flip | `mark_ask_human_resolved(session_id)` |
| MCP OAuth completed | Manual flag flip | `mark_user_mcp_connect_resolved(user_id, toolkit)` |
| Cloud PC became active | Manual flag flip | `mark_user_cloud_pc_resolved(user_id)` |
| Admin force-cancels | Manual flag flip | `mark_ask_human_resolved(session_id)` |

### 6. Remove SafetyService confirmation methods

If any custom code extends `SafetyService` and includes methods like `register_confirmation`, `get_confirmation`, or `respond_to_confirmation`, these can be removed. The module now only owns violation persistence:

```python
# Only these functions remain in safety_service.py
async def log_safety_violation(db, verdict, reason, ...)
async def update_violation_user_response(db, violation_id, approved)
```

## Compatibility notes

- Sessions created before ARCH-001 may have `waiting_for_input` set by the old imperative path. These values are harmless — on the next `_recompute_and_emit` call they will be corrected to the derived value.
- The `confirmation_request` Socket.IO event may still be emitted by older agent versions during a rolling upgrade window. Frontend code should tolerate both events during the transition.
