## Overview

Skygen records every meaningful agent action as a `ui_meta_update` system message in the chat history. The action history API queries these records and returns a structured, filterable list of what the agent did, in which mode, and at what time. This is useful for auditing automation tasks, debugging failed sessions, and showing users a clear timeline of agent activity.

---

## Data model

Each action history item is derived from a `chat_messages` row where `meta_data.type = "ui_meta_update"`. The relevant fields are extracted from `meta_data.ui_meta`:

| Field | Description |
|-------|-------------|
| `id` | Message UUID |
| `session_id` | Chat session the action belongs to |
| `action_type` | Agent action (e.g. `click`, `type`, `screenshot`, `tool`, `browse`) |
| `title` | Human-readable label for the action |
| `detail` | Additional context or description |
| `tool` | Tool name invoked (for MCP tool calls) |
| `status` | Execution status from the UI perspective |
| `timestamp` | When the action was recorded |

If `ui_meta.title` is absent, the backend applies fallback labels:

| `action_type` | Fallback title |
|---------------|----------------|
| `click` | Clicked element |
| `type` | Typed text |
| `scroll` | Scrolled page |
| `screenshot` | Took screenshot |
| `key` | Pressed key |
| `wait` | Waited |
| `drag` | Dragged element |
| `move` | Moved cursor |
| `double_click` | Double-clicked |
| `right_click` | Right-clicked |

## API reference

### List action history

```http
GET /action-history
Authorization: Bearer {token}
```

<ParamField query="session_id" type="string">
  Optional. Filter to a specific chat session.
</ParamField>

<ParamField query="date_from" type="string">
  Optional. ISO-8601 datetime lower bound (inclusive).
</ParamField>

<ParamField query="date_to" type="string">
  Optional. ISO-8601 datetime upper bound (inclusive).
</ParamField>

<ParamField query="mode" type="string">
  Optional. Filter by agent mode: `cua`, `deep_research`, `mcp`, `ask_human`.
</ParamField>

<ParamField query="limit" type="integer">
  Number of items to return. Range: 1–200. Default: 50.
</ParamField>

<ParamField query="offset" type="integer">
  Pagination offset. Default: 0.
</ParamField>

**Response — 200 OK:**

```json
{
  "items": [
    {
      "id": "msg-uuid",
      "session_id": "session-uuid",
      "action_type": "click",
      "title": "Clicked element",
      "detail": "button#submit",
      "tool": null,
      "status": "success",
      "timestamp": "2026-05-08T10:23:44.123Z"
    }
  ],
  "total": 142,
  "session_id": "session-uuid"
}
```

<ResponseField name="items" type="array">
  List of `ActionHistoryItem` objects.
</ResponseField>

<ResponseField name="total" type="integer">
  Total matching records (used for pagination).
</ResponseField>

<ResponseField name="session_id" type="string">
  Session ID filter applied, if any.
</ResponseField>

## Querying examples

### All actions for a session

```http
GET /action-history?session_id=abc123&limit=100
```

### CUA actions on a specific date

```http
GET /action-history?mode=cua&date_from=2026-05-08T00:00:00Z&date_to=2026-05-08T23:59:59Z
```

### MCP tool calls across all sessions

```http
GET /action-history?mode=mcp&limit=50&offset=0
```

## Database implementation

The endpoint uses a raw SQL query against the partial index `ix_chat_messages_ui_meta` created on `chat_messages` rows where `meta_data->>'type' = 'ui_meta_update'`. This makes queries efficient even with large message tables.

The query joins `chat_messages` with `chat_sessions` and filters by `cs.user_id` to ensure users only see their own history. Soft-deleted sessions (`deleted_at IS NOT NULL`) are excluded.

Separate `COUNT(*)` and data queries are used rather than `COUNT(*) OVER()` window function to avoid defeating the `LIMIT` optimization on large result sets.

## Notes for frontend integration

The action history powers the "Steps" panel in the chat UI, where users can see a real-time feed of what the agent is doing. Messages are appended in real time via the Socket.IO `chat_message` event — the REST endpoint is primarily for historical viewing and export.

For real-time action monitoring during an active session, subscribe to:

```
socket.on("chat_message", (msg) => {
  if (msg.meta_data?.type === "ui_meta_update") {
    // append to action timeline
  }
})
```
