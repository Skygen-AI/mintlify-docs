Skygen's agent framework automatically selects the best LLM for each step of a task based on complexity, vision requirements, and context size. You can also override the model explicitly or select a plan-level profile.

## Model Profiles (User-Facing)

From the user's perspective, Skygen exposes three **model profiles** rather than raw model names. The platform maps profiles to current best-in-class models and updates them as better models become available.

| Profile key | Display name | Use case |
|-------------|-------------|----------|
| `fast` | Skygen Fast | Lightweight tasks, quick lookups, simple automation |
| `default` | Skygen Standard | Versatile — handles most tasks well |
| `pro` | Skygen Pro | Complex multi-step reasoning, coding, long-context tasks |

Set your active profile via the billing API:

```bash
curl -X PATCH https://api.skygen.ai/billing/model-profile \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "profile": "pro" }'
```

## How Routing Works

The agent evaluates the following routing parameters for each LLM call:

```python
# RoutingParams TypedDict — fields you can influence
{
    "requires_vision": True,        # Task involves screenshots/images
    "requires_long_context": True,  # History exceeds typical context
    "requires_heavy_reasoning": True, # Multi-step planning needed
    "complexity": "high",           # "high" | "medium" | "low"
    "priority": "standard",         # "high" | "standard" | "low" | "batch"
    "force_model": "google/gemini-2.5-flash",  # Hard override
    "skip_models": ["some-model"],  # Avoid (e.g. rate-limited)
}
```

**Decision logic (simplified):**

1. If `force_model` is set, use that model directly.
2. If the task requires vision, filter to vision-capable models.
3. If the task requires long context, prefer models with large context windows.
4. If heavy reasoning is needed, prefer premium-tier models.
5. Apply circuit breaker — if a provider is experiencing errors, skip it.
6. Return the best available model and its provider.

The decision includes a `reason` string logged with each LLM call for observability.

## Provider Configuration

Each provider requires its API key set in the Agent service environment:

| Provider | Environment variable | Notes |
|----------|---------------------|-------|
| OpenRouter | `OPENROUTER_API_KEY` | Routes to Claude, Qwen, and others |
| Gemini | `GEMINI_API_KEY` | Also used for Knowledge Base embeddings |
| Groq | `GROQ_API_KEY` | Fast inference for Llama and Qwen models |
| Cerebras | `CEREBRAS_API_KEY` | Ultra-fast inference, no vision |

<Note>
  The Backend also uses `GEMINI_API_KEY` independently for Knowledge Base vector embeddings. If you deploy Backend and Agent separately, both services need the key if you enable the KB feature.
</Note>

## Circuit Breaker

Each provider has a circuit breaker that tracks failure rates. When a provider accumulates errors:

1. The circuit opens for that provider.
2. The router skips that provider and picks the next best option.
3. The circuit probes periodically to check if the provider has recovered.

This means agent tasks continue to execute even when one provider is down, without manual intervention.

## Cost Implications

Token costs are normalized into Skygen's internal token unit:

```
skygen_tokens = api_cost_usd × TOKEN_MARGIN_MULTIPLIER × TOKENS_PER_DOLLAR
             = api_cost_usd × 1.5 × 131.58
```

Premium-tier models consume tokens faster than economy-tier models. Use the `fast` profile for high-volume, lower-complexity workflows to minimize token consumption.

**Cost ranking (lowest to highest per task):**

1. Economy tier (Gemini Flash Lite, Groq Qwen 32B)
2. Balanced tier (Gemini 2.5 Flash, Llama 70B)
3. Premium tier (Claude Sonnet 4.6, Qwen 235B Thinking)

## Forcing a Specific Model

<Warning>
  Forcing a model bypasses cost optimization and circuit breaker protection. Use only for debugging or specialized workflows.
</Warning>

You can request a specific model in the message metadata. Include it in the `meta_data` field of your session create request, or configure it via the admin panel.

```bash
# The force_model is configured at the agent infrastructure level
# Contact your admin to set a per-user or per-session model override
```

For production workloads, use model profiles (`fast`, `default`, `pro`) rather than hard-coding model names — this allows the platform to upgrade models transparently.
