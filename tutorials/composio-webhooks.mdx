## Overview

Composio webhooks let external services fire your Skygen agents automatically. When an event occurs in a connected app — a new Gmail message, a new GitHub issue, a Slack mention — Composio sends a signed webhook to Skygen's `POST /mcp/webhooks` endpoint, which dispatches the event to the agent.

In this tutorial you will:
1. Connect an integration (GitHub)
2. Create a trigger that fires when a new issue is opened
3. Configure the agent's automation prompt
4. Verify end-to-end by opening a GitHub issue

---

## Prerequisites

- Skygen account with Standard plan
- GitHub connected in Skygen (follow [Connecting Integrations](/guides/connecting-integrations))
- Python 3.9+ with `httpx`
- A GitHub repository you own

## Step 1: Verify GitHub is connected

```python
import httpx

BASE_URL = "https://api.skygen.ai"
TOKEN = "your-skygen-jwt-token"
headers = {"Authorization": f"Bearer {TOKEN}"}

resp = httpx.get(f"{BASE_URL}/mcp/integrations/available", headers=headers)
items = resp.json().get("items", [])

github = next((i for i in items if i["toolkit"] == "github"), None)
if github and github["is_connected"]:
    print(f"GitHub connected — account: {github.get('account_label', 'default')}")
else:
    print("GitHub not connected. Connect at https://app.skygen.ai/settings")
```

## Step 3: Create the issue triage trigger

```python
TRIGGER_PAYLOAD = {
    "toolkit": "github",
    "trigger_name": "github_issue_created",
    "config": {
        "repo": "your-org/your-repo",
    },
    "action_prompt": """
A new GitHub issue has been created. The event payload contains the issue details.

Your task:
1. Read the issue title and body from the event payload
2. Classify the issue as one of: bug, feature_request, question, documentation, or other
3. Based on the classification, add the appropriate label to the issue using the GitHub toolkit
4. If it's a bug, add a comment: "Thank you for reporting this bug! We'll investigate and respond shortly."
5. If it's a feature request, add a comment: "Thanks for the feature idea! We've logged it for review."
6. If it's a question, search for related issues or docs and add a helpful comment pointing to relevant resources

Use the GitHub toolkit to apply labels and post comments.
"""
}

resp = httpx.post(
    f"{BASE_URL}/mcp/triggers",
    headers={**headers, "Content-Type": "application/json"},
    json=TRIGGER_PAYLOAD,
)

if resp.status_code == 201:
    trigger = resp.json()
    TRIGGER_ID = trigger["trigger_id"]
    print(f"Trigger created: {TRIGGER_ID}")
    print(f"Status: {trigger['status']}")
else:
    print(f"Error: {resp.status_code} — {resp.text}")
```

## Step 4: Understand the webhook flow

When a new GitHub issue is created:

1. GitHub sends an event to Composio.
2. Composio forwards it to Skygen's webhook endpoint:
   ```
   POST https://api.skygen.ai/mcp/webhooks
   X-Composio-Signature: sha256=<hmac>
   ```
3. Skygen verifies the HMAC signature using `COMPOSIO_WEBHOOK_SECRET`.
4. The backend parses the event and dispatches it to the agent via Socket.IO.
5. The agent runs your `action_prompt` with the event payload available as context.

The event payload passed to the agent looks like:

```json
{
  "event_type": "github_issue_created",
  "trigger_name": "github_issue_created",
  "app_name": "github",
  "payload": {
    "issue": {
      "number": 42,
      "title": "Bug: API returns 500 on empty input",
      "body": "Steps to reproduce...",
      "labels": [],
      "user": {"login": "reporter-username"}
    },
    "repository": {
      "full_name": "your-org/your-repo"
    }
  }
}
```

## Step 6: Customize the automation

### Multiple repository support

```python
# Create separate triggers for each repo
for repo in ["your-org/backend", "your-org/frontend"]:
    httpx.post(
        f"{BASE_URL}/mcp/triggers",
        headers={**headers, "Content-Type": "application/json"},
        json={
            "toolkit": "github",
            "trigger_name": "github_issue_created",
            "config": {"repo": repo},
            "action_prompt": f"Triage new issues in {repo}. Label and comment as appropriate.",
        }
    )
```

### Adding Slack notifications

Update the action prompt to also notify your team on Slack:

```python
httpx.patch(
    f"{BASE_URL}/mcp/triggers/{TRIGGER_ID}",
    headers={**headers, "Content-Type": "application/json"},
    json={
        "action_prompt": """
A new GitHub issue was created. Do the following:

1. Classify and label the issue (bug/feature/question/docs/other)
2. Post an automated comment on the issue based on its type
3. Post a notification to #engineering-alerts on Slack:
   "New [classification] issue in [repo]: [title] — [issue URL]"

Use both GitHub and Slack toolkits.
"""
    }
)
```

## Step 7: Enable and disable triggers

```python
# Disable during maintenance
httpx.patch(
    f"{BASE_URL}/mcp/triggers/{TRIGGER_ID}",
    headers={**headers, "Content-Type": "application/json"},
    json={"is_active": False},
)

# Re-enable
httpx.patch(
    f"{BASE_URL}/mcp/triggers/{TRIGGER_ID}",
    headers={**headers, "Content-Type": "application/json"},
    json={"is_active": True},
)
```

```python
    print(f"Status: {event['status']}")
    payload_preview = str(event.get("payload", {}))[:200]
    print(f"Payload: {payload_preview}")
    if event["status"] == "failed":
        print(f"Error: {event['error_message']}")
```

## Troubleshooting

**Trigger created but no events appear**  
Composio may take a few minutes to activate new triggers. Also verify the `repo` config field matches your repository's full name (`org/repo` format).

**Events showing status `failed`**  
Check the `error_message` field. Common causes:
- GitHub toolkit disconnected (reconnect in Settings)
- Repository name typo in config
- Agent hit token limit (reduce the action prompt complexity)

**`COMPOSIO_WEBHOOK_SECRET` error**  
In development, the webhook secret is optional (a warning is logged). In production, set `COMPOSIO_WEBHOOK_SECRET` in your backend ConfigMap to enable HMAC verification.

**Duplicate events firing**  
This can happen if Composio retries a failed webhook. The trigger service records each dispatch; check the event history to identify duplicates and add idempotency logic to your action prompt (e.g., "Check if the issue already has a label before applying one").
