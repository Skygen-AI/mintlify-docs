A **Cloud PC** is a persistent, pausable desktop environment powered by E2B Desktop running on Firecracker microVMs. Unlike ephemeral agent sandboxes, a Cloud PC keeps its full OS state across pause/resume cycles through incremental S3 snapshots. You create one, use it from the browser via noVNC, pause it when you are done, come back an hour or a week later, and find every window where you left it.

## What it actually is

- **Runtime**: E2B Desktop SDK on Firecracker microVMs
- **Resources**: 2 vCPU + 4 GiB RAM on the `desktop` template
- **Persistence**: custom backup layer that tars `/home/user` to S3 (capped at 500 MB), retained 30 days after termination
- **Connection**: noVNC stream over WebSocket, rendered in an iframe on the client
- **Resume latency**: 3вЂ“10 seconds from `PAUSED`, much faster than cold boot

## Relation to other concepts

|              | Cloud PC | E2B sandbox | Real device |
|--------------|----------|-------------|-------------|
| Lifecycle    | User-controlled pause and resume | Ephemeral, auto-killed | Long-lived, bound to hardware |
| Use case     | Daily automation without using your laptop | Throwaway compute for one task | Automate your actual machine |
| API surface  | `/cloud-pc/*` | `/e2b/*` | `/devices` |

A Cloud PC surfaces as a `Device` with `device_type=E2B_SANDBOX` when you call `POST /cloud-pc/link-device`, so agents can target it like any other device.

## Lifecycle endpoints

<ParamField path="POST /cloud-pc/connect" type="lifecycle">
Create a new Cloud PC or reconnect to an existing one. Request body: `{ cloud_pc_id?, name?, session_id? }`. If `cloud_pc_id` is omitted, a new one is created (subject to the `max_cloud_pcs` limit). Response includes `stream_url`, an encrypted `stream_auth_key`, resolution, and status.
</ParamField>

<ParamField path="POST /cloud-pc/disconnect" type="lifecycle">
Gracefully pause. Backs `/home/user` up to S3, then pauses the E2B sandbox. Falls back to `TERMINATED` if pause is unavailable. Reconciles outstanding billing.
</ParamField>

<ParamField path="POST /cloud-pc/heartbeat" type="lifecycle">
Keep the Cloud PC alive and deduct per-minute tokens. Send every 60 seconds while the user is viewing the stream. Returns `204 No Content`. Returns `402` if tokens are insufficient, `409` if the Cloud PC is not `ACTIVE`, `410` if the underlying sandbox is gone.
</ParamField>

<ParamField path="GET /cloud-pc/status" type="lifecycle">
Current state snapshot. Returns `stream_url` only when `ACTIVE`.
</ParamField>

<ParamField path="GET /cloud-pc/list" type="lifecycle">
List all Cloud PCs for the current user. `?include_deleted=true` includes soft-deleted entries.
</ParamField>

<ParamField path="PATCH /{cloud_pc_id}" type="lifecycle">
Rename. Body: `{ "name": "..." }` (max 100 chars).
</ParamField>

<ParamField path="DELETE /{cloud_pc_id}" type="lifecycle">
Soft-delete. Kills the E2B sandbox, keeps the S3 backup for 30 days. Returns `409` if an agent session is active.
</ParamField>

<ParamField path="POST /{cloud_pc_id}/restore" type="lifecycle">
Restore from soft-delete if the 30-day retention has not elapsed.
</ParamField>

## File management

The VNC stream does not proxy file transfers. Use presigned S3 URLs instead.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/cloud-pc/files` | List entries under `/home/user` |
| `POST` | `/cloud-pc/files/download` | Get presigned URL for a file in the sandbox |
| `POST` | `/cloud-pc/files/upload` | Get a presigned POST form to upload |
| `POST` | `/cloud-pc/files/upload-callback` | After upload, move the file from S3 into the sandbox |
| `POST` | `/cloud-pc/files/copy-attachment` | Copy a chat attachment into the Cloud PC |
| `POST` | `/cloud-pc/files/mkdir` | Create a directory |
| `DELETE` | `/cloud-pc/files` | Delete a file or directory |

All file paths are restricted to `/home/user/`. Rate-limited to 30 operations per minute per user.

## Agent control transfer

While a Cloud PC is `ACTIVE`, either the user (via noVNC) or an agent (via device actions) holds control. Transfer explicitly:

<Steps>
  <Step title="Link the Cloud PC as a device">
    `POST /cloud-pc/link-device` creates a `Device` record with `device_type=E2B_SANDBOX` named "Cloud PC". Only one linked device per user at a time.
  </Step>
  <Step title="Take control from the agent">
    `POST /cloud-pc/take-control` with `{ session_id }` pauses the agent, acquires an exclusive control lock in Redis, and logs a message to the chat.
  </Step>
  <Step title="Hand control back">
    `POST /cloud-pc/resume-ai` drains queued messages and publishes a wake-up task. `POST /cloud-pc/renew-control` extends the user's control TTL (default 120s, renewed every 50s by the frontend).
  </Step>
</Steps>

## Connect flow (client side)

<Steps>
  <Step title="Call connect">
    `POST /cloud-pc/connect` and wait for `status: "active"`. Typical time is 3вЂ“5 seconds for a new instance, 3вЂ“10 seconds to resume from pause.
  </Step>
  <Step title="Open the VNC stream">
    Render `stream_url` in an iframe. Pass the decrypted `stream_auth_key` as the query string or header expected by E2B's noVNC proxy.
  </Step>
  <Step title="Start heartbeat">
    Every 60 seconds, `POST /cloud-pc/heartbeat` with `{ is_active: true }` while the user is interacting. The backend deducts one token per minute and prevents auto-pause.
  </Step>
  <Step title="Disconnect cleanly">
    On close, `POST /cloud-pc/disconnect`. This backs up state and pauses, so resume is fast.
  </Step>
</Steps>

## Pause and resume semantics

**Preserved**: the entire `/home/user` directory (files, dotfiles, permissions), OS processes (paused, not killed), environment variables, open file descriptors, local git state.

**Lost**: clipboard, active network connections (TCP sockets time out after a few minutes of pause), X11 pixmaps (noVNC re-renders on reconnect), some daemons that exit on suspend.

**Limits**:
- Max pause duration: 30 days (S3 backup retention)
- Heartbeat interval: 60 seconds recommended
- Auto-pause on missed heartbeat: 5 minutes
- Auto-pause on max idle: 30 minutes
- Backup size cap: 500 MB

## Tier gating and billing

Cloud PC is a **paid feature**. The check lives in `plan_access_service.require_feature(db, user_id, "cloud_pc")` and runs on every `connect`. The plan's `features_config` also controls `max_cloud_pcs`:

```json
{ "cloud_pc": true, "max_cloud_pcs": 3 }
```

**Billing**: E2B costs approximately $0.1656/hour for the 2 vCPU + 4 GiB template. Skygen marks it up by 2.5Г—, so usage burns about **0.41 tokens/hour** while `ACTIVE`. Heartbeats deduct one token per minute, deduplicated in Redis so double-heartbeats do not double-bill. Pause is free for compute; S3 storage is charged separately.

## End to end

```bash
# 1. Connect (new instance)
curl -X POST https://api.skygen.ai/cloud-pc/connect \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name": "My dev PC"}'
# в†’ { "status": "active", "cloud_pc_id": "cpc_...", "stream_url": "...", "stream_auth_key": "..." }

# 2. Heartbeat loop (every 60s)
curl -X POST https://api.skygen.ai/cloud-pc/heartbeat \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"cloud_pc_id": "cpc_...", "is_active": true}'

# 3. Disconnect (pauses + backs up)
curl -X POST https://api.skygen.ai/cloud-pc/disconnect \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"cloud_pc_id": "cpc_..."}'

# 4. Resume (fast, ~5s)
curl -X POST https://api.skygen.ai/cloud-pc/connect \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"cloud_pc_id": "cpc_..."}'

# 5. Link for agent control
curl -X POST https://api.skygen.ai/cloud-pc/link-device \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"cloud_pc_id": "cpc_..."}'

# 6. Terminate
curl -X DELETE https://api.skygen.ai/cloud-pc/cpc_... \
  -H "Authorization: Bearer $TOKEN"
```

## Known rough edges

<Warning>
**No incremental backup.** Every pause tars all of `/home/user`, even if only one file changed. Large workspaces pause slowly. Consider keeping your working set trim.
</Warning>

<Note>
**500 MB backup cap.** Data beyond the cap is not backed up. Large media or datasets should stay in S3 or an external store, not in `/home/user`.
</Note>

<Note>
**Pause fallback to terminate.** If the E2B pause API fails, Skygen forcefully terminates the sandbox. Unsaved in-memory state is lost. Rare, but possible during infrastructure incidents.
</Note>
