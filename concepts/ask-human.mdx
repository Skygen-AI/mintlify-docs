## Overview

When a Skygen agent encounters a situation that requires human judgment — a missing credential, an ambiguous instruction, or a permission gate — it suspends itself and surfaces a structured prompt in the chat interface. The user provides the answer, and the agent resumes exactly where it left off. This is called the **ask-human** pattern, implemented through the hibernate/wake lifecycle in the agent runtime.

Unlike a naive polling loop, ask-human uses Redis-backed sentinel tasks. The agent pod exits its worker loop, persists state, and consumes zero CPU while waiting. A waker process monitors the sentinel store and re-enqueues the agent as soon as the wake condition is satisfied.

---

## When ask-human triggers

The agent calls `communicate(wait_for_reply=True)` when it decides it cannot proceed safely or correctly without user input. Concrete situations:

- **Missing MCP connection** — the task requires a Gmail or Slack toolkit that is not yet connected. The agent emits an `mcp_connect_prompt` card with an OAuth button.
- **Cloud PC not running** — the task needs a Cloud PC session but none is active for the user. The agent emits a `start_cloud_pc_prompt` card.
- **Text input required** — the agent needs a value it cannot infer: a password, a URL, a choice among multiple options.
- **Safety clarification** — the guardrail pipeline returned `NEED_HUMAN_CLARIFICATION` and the agent needs explicit permission before continuing.
- **Sub-agent confirmation** — a child agent (CUA Orchestrator, MCPToolsAgent) needs approval from the parent BrokerAgent to take a destructive action.

---

## What the user sees

When ask-human is active, the chat interface enters a distinctive state:

1. The message composer shows the ask-human input card instead of the normal text box.
2. The session sidebar shows a pulsing badge indicating the session is waiting for input.
3. The `session.meta_data.waiting_for_input` flag is set to `true` and broadcast via Socket.IO `session_updated` to all connected browser tabs.

The input card varies by type:

| Card type | `input.type` | User action |
|-----------|-------------|-------------|
| Free text | `input` | Type and submit a reply |
| Choice | `choice` | Select from a list of options |
| MCP connect | `mcp_connect_prompt` | Click the OAuth button for a toolkit |
| Cloud PC start | `start_cloud_pc_prompt` | Click to provision a Cloud PC session |

The backend stores each ask-human request as a `ChatMessage` row with `meta_data.type = "ask_human"` and `meta_data.ask_human_action = "request"`. This is the source of truth — `waiting_for_input` is always derived from these rows, never managed imperatively.

## How to respond

<Steps>
  <Step title="Read the agent's question">
    The chat message explains what information is needed and why. For MCP connect cards, a Connect button appears. For text prompts, a text field appears inline.
  </Step>
  <Step title="Provide your answer">
    Type your response in the ask-human composer and press Send. For OAuth cards, click the Connect button — the OAuth flow runs in the same tab and on completion the agent wakes automatically.
  </Step>
  <Step title="Wait for the agent to resume">
    After submitting, the composer returns to its normal state and the agent re-activates. The session badge clears. The agent continues from the step where it paused, injecting your answer as a user message into its context.
  </Step>
</Steps>

## The hibernate/wake lifecycle

### Hibernation

When `communicate(wait_for_reply=True)` executes, the BrokerAgent calls `HibernationController.Hibernate()` which:

1. Selects a **strategy** from the registry based on the config dict the agent emitted (e.g., `user_reply` for text input, `mcp_oauth` for OAuth-gated waits).
2. Creates a `SentinelTask` record with a unique ID, session/user/device IDs, the strategy config, and an expiry time (`expires_at = now + timeout_seconds`, default **300 seconds / 5 minutes**).
3. Writes the task to a Redis sorted set (`hibernation:tasks`) with `next_check_at` as the score.
4. Checks for a **race-window replay**: if an MCP OAuth callback arrived while the agent was still executing the LLM turn that produced this hibernate, a `pending_user_wake` key is present in Redis. If found, the task is immediately claimed and woken.
5. The agent worker loop exits — the pod thread is freed.

```python
# Simplified — see Agent/Utilities/Hibernation/Controller.py
task = await controller.Hibernate(
    session_id=session_id,
    user_id=user_id,
    device_id=device_id,
    config={"strategy": "user_reply", "timeout": 300},
)
```

### Sentinel monitoring

A background loop (the **Sentinel**) iterates the Redis sorted set on a timer. For each overdue task it evaluates the strategy's `ShouldWake()` method. If the condition is not yet met and the task has not expired, it reschedules. If expired, it publishes a timeout wake to let the agent terminate gracefully.

### Waking

Wake can be triggered by multiple paths:

| Trigger | Function called |
|---------|----------------|
| User submits a reply | `WakeSession(session_id, reason=reply_text)` |
| MCP OAuth completes | `WakeUserSessions(user_id, reason="mcp_connect_completed")` |
| Cloud PC becomes active | `mark_user_cloud_pc_resolved(user_id)` then wake |
| Sentinel timeout fires | Internal wake with `reason="timeout"` |
| Admin force-wake | `Wake(task_id)` |

All wake paths use atomic Redis `HDEL` to claim the task before publishing, preventing multiple agent pods from waking the same session simultaneously when a broadcast event is received by all pods.

```python
# Wake a specific session
woken = await controller.WakeSession(session_id, reason="User replied: yes")

# Wake all sessions for a user (e.g., after OAuth)
sessions_woken = await controller.WakeUserSessions(user_id, reason="gmail connected")
```

After a wake, the waker publishes the original payload plus the wake reason back to the BrokerAgent's input queue. The agent resumes the worker loop with the wake reason injected as context.

## ask_human_state module

The backend module `app/services/ask_human_state.py` owns the `waiting_for_input` flag lifecycle:

- `mark_ask_human_pending(session_id)` — called after a new ask_human ChatMessage row is inserted; triggers `_recompute_and_emit`.
- `mark_ask_human_resolved(session_id, kind="any")` — flips matching `ask_human_action` from `"request"` to `"answered"` and recomputes. `kind` can filter to a specific input type.
- `mark_user_mcp_connect_resolved(user_id, toolkit)` — resolves only the `mcp_connect_prompt` rows for that specific toolkit across all user sessions.
- `mark_user_cloud_pc_resolved(user_id)` — resolves all `start_cloud_pc_prompt` rows for the user.

`_recompute_and_emit` queries the database for any remaining pending ask_human rows and updates `session.meta_data.waiting_for_input`. It emits `session_updated` via Socket.IO only when the flag actually changes — avoiding spurious UI flickers.

Additionally, it emits `chat_message_meta_updated` per resolved message row so the frontend can drop ask-human styling without a reload.

---

## Timeout behavior

The default timeout is **300 seconds (5 minutes)**. This is configurable per hibernate strategy. When the timeout fires:

- The agent is woken with `reason="timeout"`.
- The agent's termination guard recognizes the timeout reason and terminates the session with an appropriate user-facing message.
- All pending ask_human rows are marked resolved via `mark_ask_human_resolved`.

If the user has multiple browser tabs open, all tabs reflect the session state change within milliseconds via Socket.IO broadcasts.

---

## Multiple pending requests

A session can have multiple pending ask_human rows simultaneously (e.g., an MCP connect prompt followed immediately by a text input). The `waiting_for_input` flag remains `true` until every pending row is resolved. The composer stays in ask-human mode for the latest unresolved request.

Resolving an MCP connect prompt does not touch sibling text-input rows — `mark_ask_human_resolved(kind="mcp_connect_prompt")` is scoped. This ensures the user is not confused by a composer that silently drops out of ask-human mode when they still owe the agent a text reply.

---

## Socket.IO events

| Event | Direction | Payload |
|-------|-----------|---------|
| `session_updated` | Server → Client | Full `ChatSessionRead` object with updated `meta_data.waiting_for_input` |
| `chat_message_meta_updated` | Server → Client | `{session_id, message_id, meta_data}` with `ask_human_action: "answered"` |
| `user_followup` | Client → Server | User reply that triggers `WakeSession` |

---

## Developer notes

When adding a new resolution path (e.g., a new OAuth provider completing), call `mark_ask_human_resolved` or the appropriate scoped variant. Do not manually flip `waiting_for_input` — it is a derived value and manual edits will be overwritten on the next recompute.

Sub-agent sentinels are automatically cleaned up by `HibernationController.CancelChildrenOf(parent_session_id)` when the parent Planner completes. This prevents phantom wake-up events from orphaned child sentinels.

## Ask-human prompt types reference

### Free text input

```python
ui_meta = {
    "type": "ask_human",
    "input": {
        "type": "input",
        "placeholder": "Type your answer here...",
        "label": "What is the target URL?",
    }
}
```

The user sees a text field in the chat composer. Their reply is injected as a user message.

### Choice / multi-choice

```python
ui_meta = {
    "type": "ask_human",
    "input": {
        "type": "choice",
        "label": "Which account should I use?",
        "options": [
            {"value": "personal", "label": "Personal Gmail"},
            {"value": "work", "label": "Work Gmail"},
        ]
    }
}
```

The user sees a list of buttons or a dropdown. The selected value is injected as a user message.

### MCP connect prompt

```python
ui_meta = {
    "type": "ask_human",
    "input": {
        "type": "mcp_connect_prompt",
        "providers": [
            {"toolkit_slug": "slack", "display_name": "Slack"}
        ],
        "message": "To post to Slack, please connect your Slack workspace."
    }
}
```

The user sees a Connect button. On OAuth completion, the session wakes automatically via `mark_user_mcp_connect_resolved`.

### Cloud PC start prompt

```python
ui_meta = {
    "type": "ask_human",
    "input": {
        "type": "start_cloud_pc_prompt",
        "message": "This task requires a Cloud PC. Click Start to provision one."
    }
}
```

The user sees a Start button. When the Cloud PC reaches ACTIVE status, the session wakes via `mark_user_cloud_pc_resolved`.

## Race condition handling

A subtle race condition can occur when an MCP OAuth callback arrives while the agent is still executing its final LLM turn — the turn that will produce the Hibernate() call. The wake event fires but finds no hibernation task to wake because the task hasn't been created yet.

The `HibernationController.Hibernate()` method handles this automatically:

1. After writing the `SentinelTask` to Redis, it calls `ConsumePendingUserWake(user_id)`.
2. If a pending wake reason was stored (from the OAuth callback that fired early), it is consumed and the task is immediately claimed and woken.
3. The agent resumes within milliseconds instead of waiting for the next sentinel poll cycle.

This ensures that even in race conditions, the user's OAuth authorization is never lost.
