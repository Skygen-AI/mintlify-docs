# Zustand Store Reference

The Skygen frontend uses a single **Zustand** store composed of **11 live domain slices**. All slices are merged at creation time via `devtools` middleware. The store is inspectable in Redux DevTools under the name `SkygenStore`.

```typescript
// store/useStore.ts
import { useStore } from 'store/useStore';

// Reading state
const user = useStore((s) => s.user);
const sessions = useStore((s) => s.sessions);

// Calling an action
const { login, logout } = useStore();

// Outside React (event handlers, WS callbacks)
const state = useStore.getState();
state.addMessage(sessionId, message);
```

The combined store type is:
```typescript
type Store = AuthSlice & SessionSlice & MessageSlice & LiveStateSlice
  & DeviceSlice & BillingSlice & UISlice & NotificationsSlice
  & HelpdeskSlice & CloudPcSlice & ScheduledTasksSlice;
```

<Warning>
Two additional slice files exist in `store/slices/` — `messageStreamSlice.ts` and `messagePlanSlice.ts` — but they are **not** spread into `useStore.ts` and are dead code. Their functionality was merged into `liveStateSlice`. Do not import from these files. See the [Dead Slices](#dead-slices) section at the bottom.
</Warning>

## authSlice (`store/slices/authSlice.ts`)

### State

```typescript
user: User | null                  // Authenticated user object from /users/me
isLoading: boolean                 // true during restoreSession, login, signup (initial: true)
isAuthenticated: boolean           // Optimistically true if auth_token exists in localStorage at module load
needsOTPSetup: boolean             // true when user.otp_secret is falsy (initial: false)
_isLoggingOut: boolean             // Guards restoreSession() against concurrent logout (initial: false)
```

**Token check at module load time:**
```typescript
// Runs synchronously before React renders — prevents login-page flash on reload
const tokenExists = hasAuthToken(); // reads localStorage.getItem('auth_token')
// isAuthenticated is initialized to tokenExists (not false)
```

<Warning>
`needsOTPSetup` is computed as `!user.otp_secret`, but `otp_secret` is not sent by the backend `UserRead` schema in all environments. Do not rely on this field until the backend contract is confirmed for your deployment.
</Warning>

### Actions

| Action | Signature | Description |
|---|---|---|
| `login` | `(email: string, password: string) => Promise<void>` | POST /auth/token → sets user, sets auth cookie, calls `setPaywallTrigger('login')`, connects `wsClient` + device WebSocket. Throws on error. |
| `signup` | `(email: string, password: string, displayName?: string, meta?: Record<string, string>) => Promise<void>` | POST /auth/register → atomic login (sets user, connects WS). Sets `needsOTPSetup: true`, calls `setPaywallTrigger('signup')`. |
| `logout` | `() => Promise<void>` | Stops device WS, disconnects wsClient, removes auth cookie, stops Cloud PC side-effects, POSTs /auth/logout, resets all per-user state across all slices. |
| `refreshUser` | `() => Promise<void>` | Re-fetches /users/me. If it fails, calls `logout()`. |
| `restoreSession` | `() => Promise<boolean>` | On app mount: calls `authApi.upgradeSession()` (cookie migration), then GET /users/me. Returns `true` on success, `false` on auth error. Network errors keep the optimistic authenticated state with a 3s retry. |
| `setNeedsOTPSetup` | `(needsSetup: boolean) => void` | Manually sets the 2FA setup flag. |

<Warning>
`logout()` spreads `...initialCloudPcState` to reset the Cloud PC slice. Any new per-user fields added to any slice must also be reset inside `logout()` — otherwise state leaks across user sessions.
</Warning>

## messageSlice (`store/slices/messageSlice.ts`)

### State

```typescript
messagesBySession: Record<string, Message[]>   // Per-session message arrays (initial: {})
isLoadingMessages: boolean                     // (initial: false)
_sendingSessionIds: Set<string>               // In-flight guard per session (prevents double-submit) (initial: new Set())
```

### Exported Types

```typescript
interface McpConnectPromptData {
  providers: Array<{
    toolkit_slug: string;
    display_name: string;
    connect_endpoint: string;
  }>;
  bundles?: Array<{
    bundle_name: string;
    providers: string[];
    reason: string;
    connect_endpoints: Array<{ toolkit_slug: string; connect_endpoint: string }>;
  }>;
}
```

### Actions

#### `loadMessages(sessionId: string) => Promise<void>`

Race-safe via per-session monotonic counter — newer calls supersede older in-flight ones.

- Merges server messages with local-only messages (ask_human prompts, pending optimistic messages)
- Preserves `display_type` and `render_data` from the store when the server omits them
- After loading: fetches plan via `sessionApi.getPlan()` unless a task is currently active with a live plan
- Clears live plan and steps if the session already has a frozen terminal plan (handles page reload after task completion)

#### `sendMessage(sessionId, content, otpCode?, deviceId?, isSandbox?, agentMode?, metaData?) => Promise<SendMessageResult>`

- Guards double-submission via `_sendingSessionIds` Set
- Detects follow-up messages (active task) — skips clearing steps/plan for follow-ups
- Creates an optimistic `temp-{timestamp}` user message immediately
- On success: replaces the temp message with the real server response
- On `payment_required + free_limit_reached`: injects a `billing_notification` system message and opens the paywall
- Pushes GTM events: `task_created` (fresh tasks only), `message_sent`
- Returns `SendMessageResult` — never throws

#### `addMessage(sessionId: string, message: Message) => void`

Deduplication logic (evaluated in order):
1. Skip if `message.id` is already in the session's message array
2. Skip recent assistant duplicates by content (within 60-second window)
3. Replace matching `temp-{id}` user message (preserves `files` metadata on the temp entry)
4. Replace optimistic `cancelled-` assistant message with the real response
5. Appends and re-sorts by timestamp to maintain chronological order

#### `updateMessage(sessionId: string, messageId: string, updates: Partial<Message>) => void`

In-place update for an existing message.

#### `cancelTask(sessionId: string, deviceId?: string, reason?: string) => Promise<void>`

- POST /agent/task/cancel
- Finalizes live state: clears typing, task status (`cancelled`), agent steps, uiMeta
- Clears `waiting_for_input` on the session
- Inserts an optimistic "Task was cancelled by user." assistant message with `display_type: 'cancelled_message'`

#### `getMessages(sessionId: string) => Message[]`

Synchronous getter — returns `messagesBySession[sessionId] ?? []`.

#### `removeMessage(sessionId: string, messageId: string) => void`

Removes a specific message by ID (used by the `message_deleted` WebSocket event).

#### `clearSessionMessages(sessionId: string) => void`

Removes all messages for a session from the store.

|---|---|---|

| `setTaskStatus` | `(sessionId: string, status: TaskStatus) => void` | Updates `taskStatusBySession`. Auto-manages `taskStartedAtBySession`: sets epoch on `isActive` becoming true (if not already set), deletes it on `isActive` becoming false. |
| `setTaskStartedAt` | `(sessionId: string, ts: number \| null) => void` | Manually seeds the elapsed timer anchor. Pass `null` to clear. |
| `updateTaskProgress` | `(sessionId: string, progress: TaskProgressParams) => void` | Maps `TaskProgressParams` fields onto `TaskStatus`. |

### Session Seal Concept

A session is "sealed" when `taskStatus.isActive === false` AND `taskStatus.status` is a terminal value. Sealed sessions reject stale in-flight WebSocket events.

```typescript
import { isSessionSealed } from 'store/slices/liveStateSlice';

const status = useStore.getState().taskStatusBySession[sessionId];
if (isSessionSealed(status)) return; // Drop stale event
```

The seal is broken when `sendMessage()` sets `isActive: true` for the next user message, or when `unsealSession()` fires on a `user_message` display_message event (covers scheduled tasks and cross-device sends).

### Paywall Trigger Flow

```
authSlice.login()  →  setPaywallTrigger('login')   [in uiSlice]
authSlice.signup() →  setPaywallTrigger('signup')  [in uiSlice]
  └─ MainLayout evaluates when billing status loads
       └─ openPaywall(reason) → uiSlice.showPaywall = true
```

Note: `openPaywall` and `setPaywallTrigger` live in **uiSlice**, not billingSlice.

## Cross-Slice Calls

Slices that need to invoke another slice's actions cast `get()` to the combined type to avoid circular imports:

```typescript
// In messageSlice:
const store = get() as unknown as FullMessageStore;
store.setTyping(sessionId, true);         // liveStateSlice
store.setTaskStatus(sessionId, {...});    // liveStateSlice
store.clearAgentSteps(sessionId);         // liveStateSlice
store.openPaywall('tokens_exhausted');    // uiSlice
store.setPlan(sessionId, payload);        // liveStateSlice

// In authSlice:
get().setPaywallTrigger('login');         // uiSlice
```

The intersection type (`FullMessageStore`, `FullStore`, etc.) is defined locally in each slice file to avoid circular imports.

## WebSocket → Store Data Flow

```
Socket.IO event
    │
    ▼

    │  (internal pub/sub, singleton)
    ▼
useChatWebSocket hook handlers        // hooks/useChatWebSocket.ts
    │  reads useStore.getState() to avoid stale closures
    ▼

    │
    ▼
Zustand state update → React re-render
```

<Info>
WebSocket handlers use `useStore.getState()` instead of closed-over state values. This avoids the React stale closure problem — the handler always reads the current store state at the time of the event, not the state captured when the `useEffect` ran.
</Info>

## Patterns and Pitfalls

<AccordionGroup>
  <Accordion title="Idempotency guards">
    `loadSessions()` and `loadDevices()` check `sessionsLoaded || isSessionsLoading` before making a network call. This prevents duplicate fetches when multiple components mount simultaneously. Use `reloadSessions()` to force a refresh.
  </Accordion>

  <Accordion title="Optimistic updates">
    `sendMessage()` adds a `temp-{timestamp}` message to the store immediately, then replaces it with the real server response. If the send fails, the temp message is updated with `status: 'error'`.

    `updateSession()` applies changes locally before the PATCH request. If the request fails, it reloads the full session list to revert.
  </Accordion>

  <Accordion title="Session-sealed guard">
    Before updating `typingBySession` or `taskStatusBySession` from a WebSocket event, the hook checks `isSessionSealed()`. Sealed sessions are in a terminal state — stale late-arriving events are dropped. The seal is broken when `sendMessage()` sets `isActive: true` for the next user message.
  </Accordion>

  <Accordion title="Plan version guard">
    `setPlan()` rejects payloads with a lower `plan_version` than the currently stored plan. This prevents stale plan snapshots from overwriting live updates during reconnects.
  </Accordion>

  <Accordion title="Message deduplication">
    `addMessage()` runs three dedup checks: by `id`, by `(role, content, timestamp)` for recent assistant messages, and temp-message replacement by `(role, content)`. The `new_message` WebSocket event also checks `displayHandledIdsRef` to skip messages already handled by `display_message`.
  </Accordion>

  <Accordion title="Cloud PC action cooldown">
    `cloudPcSlice` enforces a 3-second `actionCooldown` after every connect or disconnect. Use `clearActionCooldown()` before an intentional connect from the device picker so a recent Turn Off does not silently block it.
  </Accordion>

  <Accordion title="Logout state reset contract">
    `authSlice.logout()` explicitly resets fields from every slice. When adding new per-user fields to any slice, they MUST also be reset in `logout()`. The Cloud PC slice is reset by spreading `...initialCloudPcState`, which is why `initialCloudPcState` is exported.
  </Accordion>
</AccordionGroup>
