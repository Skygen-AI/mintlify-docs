Skygen runs on AWS EKS with Karpenter auto-scaling. Code flows through GitHub Actions вҶ’ ECR вҶ’ EKS. Two clusters separate environments (`skygen-us-dev` and `skygen-us-prod`, both `us-east-1`).

## Services

| Service | Purpose | Image | Port |
|---------|---------|-------|------|
| **backend** | FastAPI + Socket.IO HTTP API | `skygen/backend:{env}` | 8000 |
| **worker** | RabbitMQ consumer (message handlers) | `skygen/backend-worker:{env}` | 8080 health / 8001 metrics |
| **webapp** | Frontend (Nginx serving Vite build) | `skygen/webapp:{env}` | 80 |
| **agent** | Python AI agent runtime (separate repo) | `skygen/agent:{env}` | 8080 metrics |

Backend runs single-worker Uvicorn (`--workers 1 --timeout-keep-alive 300 --ws-ping-interval 30 --ws-ping-timeout 30`). Single worker because Socket.IO keeps SID maps in memory; horizontal scaling is handled by replicating pods, with Redis pub/sub as the cross-pod bus.

## Data stores

| Store | Purpose | Local | Prod |
|-------|---------|-------|------|
| **PostgreSQL** | Users, sessions, messages, devices, billing, KB embeddings | `pgvector/pgvector:pg15` container | RDS with pgvector |
| **Redis** | Socket.IO distributed state, caching, pub/sub for horizontal scaling | `redis:7-alpine` | ElastiCache |
| **RabbitMQ** | Task queues (`SEND_MESSAGE_TO_AGENT_QUEUE`, `ANSWER_AGENT_QUEUE`) | `rabbitmq:3-management-alpine` | Amazon MQ (`mq.t3.micro`, SINGLE_INSTANCE per env) |
| **S3 / MinIO** | Screenshots, attachments, Cloud PC backups | MinIO container | AWS S3 |

S3 buckets (prod):

- `skygen-screenshots-us` вҖ” main artifacts
- `skygen-files-us` вҖ” user attachments
- `skygen-cloudpc-profiles` вҖ” Cloud PC snapshots (30-day retention)

## Clusters

### Dev вҖ” `skygen-us-dev`

- Namespace: `skygen-agent` (backend + worker + agent), `monitoring` (Prometheus/Grafana/Loki)
- Replicas: fixed (backend 2, worker 2, agent 2)
- Ingress: shared ALB group `skygen-dev-shared` (preview environments reuse it)
- Domain: `dev.skygen.ai` вҶ’ ALB вҶ’ `backend-service:8000`

### Prod вҖ” `skygen-us-prod`

- Namespace: `skygen-prod`
- Replicas: HPA-managed
  - Backend: 3 вҶ’ 30 (CPU 70%, memory 80%)
  - Agent: 3 вҶ’ 100
  - Worker: 2 вҶ’ 20
- Fargate-compatible: DaemonSets removed (node-exporter), Promtail as StatefulSet
- Domains: `api.skygen.ai`, `app.skygen.ai`

## CI/CD

**GitHub Actions** workflow triggers on push:

- Push to `develop` вҶ’ build + push to ECR вҶ’ deploy to dev cluster
- Push to `main` вҶ’ build + push вҶ’ deploy to prod
- User cannot push directly to `main` or `develop` (protected branches, PR required)

Images go to ECR: `118183257315.dkr.ecr.us-east-1.amazonaws.com/skygen/{backend,backend-worker,webapp,agent}:{dev,prod}`.

Deploy uses `kubectl rollout` with `RollingUpdate` strategy, zero downtime. An init container on every deploy runs `alembic upgrade heads && python -m scripts.create_admin` to ensure schema and default admin exist.

## Local development

`docker-compose-local.yml` at the monorepo root spins up: PostgreSQL, Redis, RabbitMQ, MinIO, MailHog. Backend runs with `uvicorn app.main:socket_app --reload --port 8000`.

```bash
# Bring up dependencies
docker compose -f docker-compose-local.yml up -d

# Backend
cd Backend && poetry install && poetry run uvicorn app.main:socket_app --reload

# WebApp
cd WebApp && npm install && npm run dev
```

Reset DB from scratch:

```bash
docker compose -f docker-compose-local.yml down
docker volume rm monorepo_postgres_data
docker compose -f docker-compose-local.yml up -d
```
