## Overview

Skygen's observability stack runs in the `monitoring` namespace on both dev (`skygen-us-dev`) and prod (`skygen-us-prod`) clusters. It consists of:

- **Prometheus** — scrapes metrics from all backend and agent pods
- **Grafana** — dashboard visualization at `grafana-dev.skygen.ai` (dev) / `grafana.skygen.ai` (prod)
- **Loki** — log aggregation via Promtail
- **Alertmanager** — routes alerts to Telegram

## Prometheus metrics

### Backend metrics

The backend exposes Prometheus metrics at `GET /metrics` (enabled by `ENABLE_METRICS: "true"` in the configmap). Metrics are defined in `Backend/app/monitoring/metrics.py`.

#### User metrics

| Metric | Type | Description |
|--------|------|-------------|
| `skygen_users_total` | Gauge | Total registered users |
| `skygen_users_active` | Gauge | Active users (`is_active=true`) |
| `skygen_users_online` | Gauge | Users connected via WebSocket |
| `skygen_users_dau` | Gauge | Daily active users (last 24h) |
| `skygen_users_wau` | Gauge | Weekly active users (last 7d) |
| `skygen_users_mau` | Gauge | Monthly active users (last 30d) |
| `skygen_users_new_today` | Gauge | Users registered today |

#### Agent metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `skygen_agent_requests_total` | Counter | `status` | Total requests sent to agent |
| `skygen_agent_request_duration_seconds` | Histogram | — | Request processing duration. Buckets: 0.1s to 300s |
| `skygen_devices_connected` | Gauge | — | Devices currently connected |

#### HTTP metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `skygen_http_errors_total` | Counter | `status_code`, `method`, `path` | HTTP 5xx errors |
| `skygen_http_requests_by_type_total` | Counter | `caller_type`, `method` | Requests by `user` / `internal` / `anonymous` |
| `skygen_http_requests_by_category_total` | Counter | — | Requests by functional category |

#### Queue metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `skygen_queue_messages` | Gauge | `queue_name` | Messages in RabbitMQ queue |
| `skygen_queue_consumers` | Gauge | `queue_name` | Active consumers per queue |

#### WebSocket metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `skygen_websocket_connections` | Gauge | `connection_type` | Active Socket.IO connections |
| `skygen_websocket_messages_sent_total` | Counter | `connection_type`, `message_type` | Outbound messages |
| `skygen_websocket_messages_received_total` | Counter | `connection_type`, `message_type` | Inbound messages |

#### Session metrics

| Metric | Type | Description |
|--------|------|-------------|
| `skygen_chat_sessions_total` | Gauge | Total chat sessions |
| `skygen_chat_messages_total` | Gauge | Total messages (by role) |

#### Worker metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `skygen_worker_errors_total` | Counter | `error_type`, `message_type` | Worker processing errors |
| `skygen_worker_messages_processed_total` | Counter | `message_type`, `status` | Messages processed by workers |

### Agent metrics

The agent exposes metrics at port 8080 via `GET /metrics`, defined in `Agent/metrics.py`. Key metrics:

| Metric | Type | Description |
|--------|------|-------------|
| `skygen_agent_llm_calls_total` | Counter | LLM API calls by model and status |
| `skygen_agent_llm_latency_seconds` | Histogram | LLM call latency |
| `skygen_agent_task_duration_seconds` | Histogram | Full task execution duration |
| `skygen_agent_task_wait_time_seconds` | Histogram | Time spent waiting for user input |
| `skygen_agent_cost_usd_total` | Counter | Total LLM spend in USD |

## Prometheus scraping

Backend pods announce themselves via pod annotations:

```yaml
annotations:
  prometheus.io/scrape: "true"
  prometheus.io/port: "8000"
  prometheus.io/path: "/metrics"
```

Agent pods similarly expose port 8080 with the `/metrics` path.

## Grafana dashboards

Dashboards are stored as Kubernetes ConfigMaps with label `grafana_dashboard: "1"`. Grafana auto-discovers and imports them. Dashboard files are in `Backend/k8s/monitoring/`.

Access:
- Dev: `https://grafana-dev.skygen.ai`
- Prod: `https://grafana.skygen.ai`

Credentials are stored in AWS Secrets Manager and synced via ExternalSecrets.

## Loki log aggregation

Loki is installed via `loki-stack` Helm chart in the `monitoring` namespace. Promtail runs as a DaemonSet on all EC2 nodes and ships pod logs to Loki.

Query logs in Grafana Explore:

```logql
{namespace="skygen-agent", app="backend"} |= "ERROR"
```

```logql
{namespace="skygen-agent", app="agent"} |= "task_id" | json | task_id="abc123"
```

<Note>
  The `loki.datasource.isDefault` Helm value has a bug: due to the Go template `default` function, it always renders `true`. Pass `--set loki.datasource.isDefault=false` explicitly when installing to avoid a duplicate datasource crash.
</Note>

## Alertmanager

Alerts route to Telegram via the alertmanager configuration. Receiver indices:
- `[0]` — null receiver (silenced)
- `[1]` — Telegram for non-critical alerts
- `[2]` — Telegram for critical alerts

Dev Telegram chat ID: `-5147725776`

Configure via env:
- `ALERTMANAGER_TELEGRAM_BOT_TOKEN`
- `ALERTMANAGER_TELEGRAM_CHAT_ID`

## Installing the monitoring stack

```bash
# From Backend/k8s/monitoring/
./install-monitoring.sh

# Or manually:
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update

# kube-prometheus-stack (Prometheus + Grafana + Alertmanager)
helm install kube-prom prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  -f monitoring/prometheus-values.yaml

# loki-stack (Loki + Promtail)
helm install loki grafana/loki-stack \
  --namespace monitoring \
  --set loki.datasource.isDefault=false \
  -f monitoring/loki-values.yaml
```
