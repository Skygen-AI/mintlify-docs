Skygen runs on AWS EKS in the `us-east-1` region. This guide covers the Kubernetes deployment structure, how to trigger deployments, and how to operate the cluster day-to-day.

## Cluster Overview

| Cluster | Namespace | Purpose |
|---------|-----------|---------|
| `skygen-us-dev` | `skygen-agent` | Development / staging |
| `skygen-us-prod` | `skygen-prod` | Production |

Each cluster uses **Karpenter** for node autoprovisioning with dedicated node pools per workload type.

## Node Pools

| Pool | Taint | Workloads |
|------|-------|-----------|
| `role=backend` | `role=backend:NoSchedule` | Backend API pods |
| `role=worker` | `role=worker:NoSchedule` | RabbitMQ consumer workers |
| `role=agent` | `role=agent:NoSchedule` | AI agent service pods |

Pods use `nodeSelector` + `tolerations` to land on the correct pool. This isolates agent workloads (which can spike CPU during LLM calls) from the API serving traffic.

## Deployed Services

```yaml
# Backend deployment summary (k8s/backend/deployment.yaml)
name: backend
namespace: skygen-agent
replicas: 2  # dev; HPA-managed in prod
```

## ConfigMap Structure

Non-sensitive configuration is stored in `backend-config` ConfigMap. Sensitive values come from ExternalSecrets. Example dev values:

```yaml
# Backend/k8s/backend/configmap.yaml
data:
  FRONTEND_URL: "https://app-dev.skygen.ai/app"
  API_BASE_URL: "https://dev.skygen.ai"
  SEND_MESSAGE_TO_AGENT_QUEUE: "content_dev"
  ANSWER_AGENT_QUEUE: "answer_agent_dev"
  AUTH_COOKIE_NAME: "skygen_session_dev"
  AUTH_COOKIE_DOMAIN: ".skygen.ai"

  KB_ENABLED: "true"
  BILLING_ENABLED: "true"
  ENABLE_METRICS: "true"
  ENVIRONMENT: "dev"
```

## Secrets Management

Secrets flow from **AWS Secrets Manager** → **ExternalSecrets** → **K8s Secret** → Pod environment.

```yaml
# External secret example
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: backend-secrets-us
  namespace: skygen-agent
spec:
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: backend-secrets-us
  data:
    - secretKey: SECRET_KEY
      remoteRef:
        key: skygen/dev/backend
        property: SECRET_KEY
```

Secrets are organized by region:
- `backend-secrets-us` — US-region secrets (Stripe keys, Telegram tokens, Whisper)
- `backend-secrets-eu` — EU-region secrets (legacy ECS secrets)

## GitHub Actions CI/CD

### Deployment workflows

Deployments are triggered via **`workflow_dispatch`** — there is no auto-deploy on merge.

```bash
# Deploy Backend to dev
gh workflow run k8s-dev.yml -f branch=develop

# Deploy Backend to production
gh workflow run k8s-prod.yml -f branch=main

# Deploy Agent to dev (run from Agent repo)
gh workflow run k8s-dev.yml -f branch=develop

# Coordinated deploy (Backend + Agent simultaneously)
gh workflow run coordinated-deploy-dev.yml
gh workflow run coordinated-deploy-prod.yml
```

Or use the GitHub Actions UI: **Actions** tab → select workflow → **Run workflow** → choose branch.

<Warning>
  You cannot push directly to `main` or `develop` branches. Create a PR and merge through the review process.
</Warning>

### WebApp deployment

WebApp (in `ui-shared-tauri`) has a push-triggered deploy:
- `push: develop` → auto-triggers `deploy-dev.yml` to ECS in eu-north-1
- No manual trigger needed for WebApp dev deployments

## Rolling Deployments

Backend uses `RollingUpdate` strategy:

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

Zero downtime during deploys. The init container (migration) runs before new pods start serving traffic.

## Horizontal Pod Autoscaler

Production uses HPA for Backend pods:

```yaml
# k8s/backend/hpa.yaml
minReplicas: 2
maxReplicas: 10
metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

## Database Migrations

Migrations run automatically in the init container on every deploy. The init container runs:

```bash
alembic upgrade heads && python -m scripts.create_admin
```

For manual migration runs:

```bash
kubectl exec -n skygen-agent deploy/backend -- alembic upgrade heads
```

To create a new migration locally:

```bash
cd Backend
alembic revision --autogenerate -m "add_column_foo_to_bar"
```

<Warning>
  PgBouncer runs in `transaction` pool mode. Never use `CREATE INDEX CONCURRENTLY`, `REINDEX CONCURRENTLY`, or `VACUUM` in migrations — they will fail. Use plain `CREATE INDEX` or run these commands directly against the Postgres endpoint (bypassing PgBouncer).
</Warning>

## Monitoring Setup

Monitoring runs in the `monitoring` namespace via Helm:

```bash
# Install Prometheus + Grafana
helm upgrade --install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --values Backend/k8s/monitoring/values.yaml

# Install Loki (log aggregation)
helm upgrade --install loki-stack grafana/loki-stack \
  --namespace monitoring \
  --set loki.datasource.isDefault=false  # Required! Helm gotcha — must be explicit
```

<Note>
  Pass `--set loki.datasource.isDefault=false` explicitly. The Loki chart has a Helm `default` function bug that always renders this `true` if you rely on the default, causing a duplicate Grafana datasource crash.
</Note>

### Grafana dashboards

Dashboards are stored as ConfigMaps with the label `grafana_dashboard: "1"`:

```bash
# Backend/k8s/monitoring/dashboards/
kubectl apply -f Backend/k8s/monitoring/dashboards/
```

Grafana auto-discovers ConfigMaps with the label and imports them.

### Alertmanager

Alerts route to Telegram. Channel indices in the config:
- `[0]` = null receiver
- `[1]` = `telegram-{env}` (warnings)
- `[2]` = `critical-{env}` (critical)

## Useful kubectl Commands

```bash
# View Backend logs
kubectl logs -f deploy/backend -n skygen-agent

# View Agent logs
kubectl logs -f deploy/agent-service -n skygen-agent

# Check pod status
kubectl get pods -n skygen-agent

# Describe a failing pod
kubectl describe pod <pod-name> -n skygen-agent

# Exec into Backend pod
kubectl exec -it deploy/backend -n skygen-agent -- bash

# Check HPA status
kubectl get hpa -n skygen-agent

# View Backend configmap
kubectl get configmap backend-config -n skygen-agent -o yaml

# Scale Backend manually (dev)
kubectl scale deployment backend --replicas=3 -n skygen-agent

# Roll out a restart (picks up new ExternalSecret values)
kubectl rollout restart deployment/backend -n skygen-agent
```

## PodDisruptionBudget

```yaml
# k8s/backend/pdb.yaml
minAvailable: 1
```

Ensures at least one Backend pod is always available during node maintenance or rolling deploys.

## ServiceAccount and IRSA

The Backend pod uses an IAM Role for Service Accounts (IRSA) with the `backend-sa` service account:

```yaml
serviceAccountName: backend-sa
```

The role grants access to:
- S3 buckets (screenshots, attachments, Cloud PC profiles)
- SES (email sending via `aioboto3 sesv2`)
- Secrets Manager (for AWS-native credential fetching)

No access keys in pods — credentials come from the IRSA token automatically.
