## Overview

The Skygen WebApp is a React 18 + TypeScript single-page application built on Vite. All pages live in `WebApp/ui-shared-tauri/src/pages/` and are shared between the web app and the macOS/Windows desktop app (Tauri). Routing is handled by React Router v6 and defined in `App.tsx`. Pages consume shared state from the Zustand store (`useStore`) and communicate with the backend via HTTP clients and Socket.IO.

The web app is served under the `/app` base path (`bootstrap({ basePath: '/app' })`). Routes listed below are relative to that base.

## HomeUpdated (Home Page)

**Route:** `/` (unauthenticated → redirects to `/login`; authenticated → renders HomeUpdated) or `/home`  
**File:** `HomeUpdated.tsx`

The main landing page for authenticated users. Displays the task input, quick action chips, and integration connection CTAs.

**Components used:**
- `TaskInput` — main task entry field with file attachment support
- `ChipButton` — quick action category selector (Productivity, Research, Health, More)
- `SuggestionItem` — suggestion pills that pre-fill the task input
- `ConnectorsModal` — integration management drawer
- `ConnectMcpModal` — OAuth connection flow modal
- `AddDeviceModal` / `ChooseDeviceModal` — device selection for CUA tasks
- `DeviceAppModal` — app selector for device automation
- `ConnectToolsCta` — prompt to connect integrations when none are connected
- `DropOverlay` — full-page file drag-and-drop overlay

**Store state read:**
- `cloudPcStatus` via `useComputerUse()` — Cloud PC readiness state
- `billingStatus` via `useBilling()` — plan access and token balance
- `planAccess` via `usePlanAccess()` — feature access flags
- `connectors` via `useConnectors()` context — connected MCP integrations

**Key behavior:**
- On mount, loads landing stats and prompt stats for suggestion generation
- Quick action chips filter the suggestion pills by category
- `useOAuthReturn()` hook strips OAuth callback params from URL and refreshes connector state
- `useStripeCheckoutReturn()` hook handles Stripe redirect after subscription

## ChatPage

**Route:** `/chat/:sessionId` — also `/chat` (no sessionId, renders empty chat) and `/app` (alias)  
**File:** `ChatPage.tsx`

The core agent interaction interface. Shows the message history, live progress indicators, and the ask-human composer.

**Components used:**
- `MessageItem` — renders each chat message (user, assistant, system, ask_human cards)
- `ChatLiveProgress` — real-time "agent is thinking" indicator with action steps
- `InputPanel` — message composer with attachments, voice input, and ask-human mode
- `ChatModals` — connector and device modals scoped to chat context
- `DropOverlay` — file drag-and-drop

**Hooks used:**
- `useChat()` — session loading, message streaming, send/cancel
- `useBilling()` — token balance gate
- `useOAuthReturn()` — handle same-tab OAuth redirect
- `useChatSend()` — message submission logic
- `useChatScroll()` — auto-scroll to latest message
- `useChatConnectors()` — connector state scoped to this session
- `useChatDeviceState()` — device selection and status for this session
- `useSessionFiles()` — emitted files and attachment preview
- `useFullPageDrop()` / `useFullPagePaste()` — file input handlers

**Store state read:**
- `messages[sessionId]` — message list for the active session
- `sessions[sessionId]` — session metadata including `meta_data.waiting_for_input`
- `taskStatus` — agent running / paused / idle state
- `isFileManagerOpen` — file panel toggle

**Key behavior:**
- URL param `sessionId` drives the active session
- When `waiting_for_input = true`, the `InputPanel` switches to ask-human mode
- Socket.IO `session_updated` keeps `waiting_for_input` in sync across tabs
- `taskStatus.isActive` is restored on reload from `waiting_for_input` flag

## LibraryPage

**Route:** `/library`  
**File:** `LibraryPage.tsx`

Displays the use-case library with tabs for each category and cards for individual prompts.

**Components used:**
- `TabGroup` / `Tab` — category tab selector
- `UseCaseCard` — individual library card with title, badge, and launch button

**State:**
- Local state: `types`, `cards`, `activeType`, loading/error flags
- `useBilling()` — used to check plan access before launching a card

**Key behavior:**
- On mount, calls `libraryApi.getTypes()` to load categories
- When `activeType` changes, calls `libraryApi.getCardsByType(slug)`
- Clicking a card navigates to `/chat` with the prompt pre-filled

## PricingPage

**Route:** `/subscriptions` (was documented as `/pricing` — **that route does not exist**)  
**File:** `PricingPage.tsx`

Subscription management and plan selection. Shows available plans and the user's current subscription status.

**Components used:**
- `PricingCard` — individual plan display with pricing, features, and subscribe button
- `AddCardModal` — Stripe card input modal for first-time payment
- `ConfirmUpgradeModal` — plan change confirmation dialog

**Hooks used:**
- `useBilling()` — `plans`, `pricing`, `status`, `subscribe`, `changePlan`, `createCheckoutSession`
- `useStripeCheckoutReturn()` — handle Stripe Checkout redirect back to app
- `useWebSocket()` — subscribe to `subscription_updated` real-time events

**State:**
- `isPricingLoading`, `isPlansLoading`, `isStatusLoading`
- `showAddCardModal`, `showConfirmModal`, `selectedPlan`
- `hasPaymentMethod` — drives which subscribe flow to use
- `hasActiveSubscription` — shows current plan vs available plans

**Key behavior:**
- Deep-link: `?plan_id=...` in URL or sessionStorage auto-triggers the upgrade flow on load
- Stripe Checkout creates a new browser session; after return, `useStripeCheckoutReturn` polls for subscription activation

## LoginPage

**Route:** `/login`  
**File:** `LoginPage.tsx`

Email + password authentication. Redirects authenticated users to `/` on load. Handles Google OAuth error codes via `?google_error=` query param.

**Components used:**
- `SignInForm` — email + password form
- `OTPDialog` — shown post-login when 2FA is enabled
- `ForgotPasswordModal` — password reset request flow
- `HeroCard` — marketing hero panel (desktop viewport)
- `Squares` — animated grid background

**Hooks used:**
- `useAuth()` — `login`, `isAuthenticated`, `isLoading`
- `useVisualViewport()` — adjusts layout on mobile keyboard open
- `useNavigate()`, `useLocation()`

**Key behavior:**
1. `login()` from `authSlice` calls `POST /auth/token` and stores token + sets cookie.
2. After login, a second call to `GET /users/me` checks `user.otp_secret === true`.
3. If 2FA is enabled → shows `OTPDialog`. Correct code → `POST /auth/verify-otp`.
4. After successful auth, checks `billingApi.getStatus()`:
   - `can_access=false` + paywall reason → opens PaywallModal
   - Otherwise → navigates to `/` (or `?redirect=` path)
5. URL query params are forwarded to the post-login destination (supports landing page deep-links like `?usecaseId=...&prompt=...`).
6. Prompt stats are reported via `promptStatsApi` for GA attribution.

## RegisterPage

**Route:** `/register`  
**File:** `RegisterPage.tsx`

New account registration. Supports invite code entry when `REGISTRATION_MODE=invite_only`.

**Components used:**
- `SignInForm` — used in register mode (shows display name + invite code fields)
- `HeroCard`, `Squares` — same as LoginPage

**Hooks used:**
- `useAuth()` — `signup`, `isAuthenticated`, `isLoading`
- `useVisualViewport()`

**Key behavior:**
1. Collects GA tracking IDs from `_ga` cookie via `getGAClientId()` and `getGASessionId()`.
2. Passes UTM params + GA data as `meta` to `signup()` for analytics attribution.
3. `signup()` calls `POST /auth/register` and stores token + sets cookie.
4. After registration, checks billing and navigates to `/` (or opens PaywallModal).
5. If already authenticated on mount, redirects to `/` immediately.

## GoogleCallbackPage

**Route:** `/auth/google/callback`  
**File:** `GoogleCallbackPage.tsx`

Handles the Google OAuth sign-in callback. The backend sets the `skygen_session` cookie and redirects to `/app/home` (web) or appends `?token=&is_new=` to the redirect URI for native clients.

**Components used:**
- `LoadingScreen` — shown while the callback is processing

**Hooks used:**
- `useAuth()` — `restoreSession`
- `useNavigate()`

**Key behavior:**
1. Reads `token` and `is_new` from URL query params.
2. Stores `token` in `platform.storage` (localStorage on web, secure Tauri storage on desktop).
3. Calls `restoreSession()` to hydrate the auth state from the stored token.
4. For new users (`is_new=true`): sends registration meta (UTM + GA IDs) and prompt stats for attribution.
5. Checks billing status:
   - `can_access=false` + paywall reason → opens PaywallModal then navigates home
   - Otherwise → navigates to `/`
6. On failure (no token in URL), navigates to `/login`.

## OAuthCallbackPage

**Route:** `/oauth/callback` (was documented as `/mcp/oauth/callback` — **that route does not exist**)  
**File:** `OAuthCallbackPage.tsx`

Handles Composio OAuth redirects after connecting integrations (Gmail, Slack, GitHub, etc.). Reads the `status` and `appName` query params, updates the connector state, and navigates back to the `return_to` path embedded in the signed state token.

| `/oauth/callback` | `OAuthCallbackPage` | No |
| `/app` | `ChatPage` (no session) | Yes |

| `/notifications` | `NotificationsPage` | Yes |
| `/library` | `LibraryPage` | Yes |

| `/helpdesk/kb` | `HelpdeskPage` (KB tab) | Yes |
| `/helpdesk/kb/:slug` | `HelpdeskPage` (KB article) | Yes |

| `/ui-kit` | `UiKitPlayground` | No |
| `*` | Redirect to `/` | — |

<Info>
All routes are relative to the `/app` base path configured in `bootstrap({ basePath: '/app' })`. Full URLs are therefore `https://app.skygen.ai/app/chat/:sessionId`, etc.
