# WebApp Architecture

The Skygen WebApp follows a **Platform Shell Pattern**: the browser app (`WebApp/`) is a thin configuration wrapper (~20 lines) that bootstraps a shared UI library (`ui-shared-tauri/`) which contains all real application code. This separation allows the same UI codebase to run inside a Tauri desktop app and the browser without conditional imports.

## Repository Layout

```
WebApp/
├── src/                          # Web wrapper (~3 files)
│   ├── main.tsx                  # Entry point — bootstrap + render
│   ├── config/api.ts             # VITE_BACKEND_URL / VITE_WS_URL / VITE_STRIPE_PUBLISHABLE_KEY
│   └── platform/webPlatformAdapter.ts   # IPlatformAdapter for browser
│
└── ui-shared-tauri/src/          # All real UI code (git submodule)
    ├── api/
    │   ├── http/                 # HTTP API modules (one file per domain)
    │   └── websocket/            # Socket.IO singleton client
    ├── store/
    │   ├── useStore.ts           # Zustand store composition
    │   └── slices/               # 11 domain slices
    ├── components/               # Reusable React components
    ├── pages/                    # Page-level components
    ├── hooks/                    # Custom React hooks
    ├── types/                    # TypeScript type definitions
    ├── contexts/                 # React contexts
    ├── utils/                    # Pure utilities
    └── init/bootstrap.ts         # Bootstrap + global config accessors
```

<Info>
The `ui-shared-tauri/` directory is a **git submodule**. When modifying UI code, you must commit in both the submodule repository and the parent `WebApp/` repository.
</Info>

## Bootstrap Lifecycle

The app must be initialized before any React rendering starts.

```typescript
// WebApp/src/main.tsx
bootstrap({
  apiConfig: {
    BACKEND_URL: import.meta.env.VITE_BACKEND_URL || 'https://dev.skygen.ai',
    WS_URL:      import.meta.env.VITE_WS_URL      || 'wss://dev.skygen.ai',
    STRIPE_PUBLISHABLE_KEY: import.meta.env.VITE_STRIPE_PUBLISHABLE_KEY || '',
  },
  platform: webPlatformAdapter,
  basePath: '/app',
});

ReactDOM.createRoot(document.getElementById('root')!).render(<App />);
```

`bootstrap()` stores the `apiConfig` and `platform` adapter in module-level variables inside `init/bootstrap.ts`. All other modules call `getBackendUrl()`, `getWsUrl()`, and `getPlatform()` to access them — there is no prop-drilling or React context for these.

### Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `VITE_BACKEND_URL` | `https://dev.skygen.ai` | REST API base URL |
| `VITE_WS_URL` | `wss://dev.skygen.ai` | Socket.IO server URL |
| `VITE_STRIPE_PUBLISHABLE_KEY` | `""` | Stripe.js init key |

## Platform Adapter

The `IPlatformAdapter` interface abstracts platform capabilities. The web implementation uses `localStorage` for storage and `window.location.href` for OAuth:

```typescript
// WebApp/src/platform/webPlatformAdapter.ts
export const webPlatformAdapter: IPlatformAdapter = {
  platform: 'web',
  capabilities: {
    hasDesktopAutomation: false,   // No device WebSocket on web
    hasSecureStorage: false,        // Uses localStorage, not OS keychain
    hasOverlay: false,
    canOpenOAuthWindow: true,
  },
  storage: {
    get:    async (key) => localStorage.getItem(key),
    set:    async (key, value) => localStorage.setItem(key, value),
    remove: async (key) => localStorage.removeItem(key),
  },
  openOAuthWindow: async (url) => { window.location.href = url; },
  desktop: undefined,              // No device WebSocket for web
  analytics: {
    identify: (userId) => window.dataLayer.push({ event: 'identify', user_id: userId }),
    track:    (event, props) => window.dataLayer.push({ event, ...props }),
  },
};
```

The desktop (Tauri) adapter implements `desktop.startDeviceWebSocket()` and uses OS-level secure storage. Feature-gating is done in `authSlice` via `platform.capabilities.hasDesktopAutomation`.

<AccordionGroup>
  <Accordion title="api/ — HTTP and WebSocket clients">
    **`api/http/httpClient.ts`** — Base `fetch` wrapper. Injects auth headers, handles `credentials:'include'`, throws typed `HttpClientError` on non-2xx.

    **`api/http/authApi.ts`** — Auth endpoints: login, register, logout, 2FA, token refresh, session upgrade.

    **`api/http/sessionApi.ts`** — Session CRUD, message history, message send, plan fetch, task cancel/status.

    **`api/http/billingApi.ts`** — Billing status, plans, Stripe setup, subscriptions.

    **`api/http/deviceApi.ts`** — Device registration, listing, deletion.

    **`api/http/cloudPcApi.ts`** — Cloud PC lifecycle, file manager, control transfer.

    **`api/http/notificationsApi.ts`** — Notification listing and read marking.

    **`api/http/scheduledTasksApi.ts`** — Scheduled task CRUD + run history.

    **`api/websocket/wsClient.ts`** — `SocketIOClient` singleton. Connects to `/user` namespace, manages all event subscriptions, deduplicates durable events by `notification_id`.
  </Accordion>

  <Accordion title="store/ — Zustand state management">
    Single Zustand store composed of 11 slices. See [State Management](/webapp/state-management) for details.

    | Slice | Responsibility |
    |---|---|
    | `authSlice` | User auth, session restore, login/logout lifecycle |
    | `sessionSlice` | Chat session CRUD |
    | `messageSlice` | Message history, send, cancel, optimistic updates |
    | `liveStateSlice` | Task status, typing indicators, agent steps, plan |
    | `deviceSlice` | Device list, selection, status updates |
    | `billingSlice` | Billing status, plans, subscription mutations |
    | `cloudPcSlice` | Per-session Cloud PC state, heartbeat lifecycle |
    | `uiSlice` | Theme, sidebar, confirmations, paywall, modals |
    | `notificationsSlice` | Admin notifications |
    | `helpdeskSlice` | Support ticket state |
    | `scheduledTasksSlice` | Scheduled task CRUD |
  </Accordion>

  <Accordion title="hooks/ — Custom React hooks">
    **`useChatWebSocket`** — Subscribes to all Socket.IO events for a chat session. Contains a 120s watchdog timer that REST-polls `/agent/task/{id}/status` every 15s while a task is active to unstick a stale spinner.

    **`useChatSend`** — Wraps `messageSlice.sendMessage()` with device/mode selection logic.

    **`useAuth`** — Exposes auth state from store.

    **`useDevices`** — Device list with load-on-first-use.

    **`useCloudPCActions`** — Connect/disconnect/take-control for Cloud PC.

    **`useCloudPCEvents`** — Subscribes to `cloud_pc_status_changed`, `cloud_pc_control_changed`, etc.

    **`useMainLayoutInit`** — Runs the initial data loading sequence after auth: sessions, devices, billing status, notifications.

## Data Flow Diagram

```
User action
    │
    ▼
React Component
    │  reads state via useStore() / custom hook
    ▼
Zustand Store (useStore)
    │  dispatches actions
    ▼
Store Slice action (e.g. messageSlice.sendMessage)
    │  calls API module
    ▼
HTTP API module (e.g. sessionApi.sendMessage)
    │  fetch() with credentials:'include' + Bearer header
    ▼
Backend REST API

 Backend → Socket.IO event
    │
    ▼
wsClient (SocketIOClient singleton)
    │  internal pub/sub
    ▼
useChatWebSocket hook handlers
    │  useStore.getState().<action>()
    ▼
Zustand Store
    │  triggers re-render
    ▼
React Components
```

## Auth Cookie Migration

The backend is actively migrating from pure Bearer-token auth to `skygen_session` HttpOnly cookies. The frontend handles both:

1. **First login/register** — backend sets `skygen_session` cookie AND returns JWT in body. Both are stored.
2. **Session restore** — `authApi.upgradeSession()` converts any legacy localStorage token to a cookie before every `/users/me` call.
3. **HTTP requests** — `httpClient` uses `credentials:'include'`. Backend prefers cookie when present; falls back to Bearer for native clients.
4. **Socket.IO** — `withCredentials: true` on the handshake + `auth.token` fallback for native.
5. **Phase 2** — `/auth/upgrade-session` returns `410 Gone` once migration is complete; frontend auto-purges stale localStorage tokens on 410.
