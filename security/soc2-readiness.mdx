This page is Skygen's internal SOC 2 readiness plan. It lists what we already have, where we fall short, and the exact sequence of work to reach a Type 1 attestation and then a Type 2 observation window.

<Note>
**Target:** SOC 2 Type 1 report in ~3–4 months, then 6–12 month observation window for Type 2. Start scope: **Security** Trust Service Criterion only (Availability, Confidentiality, Processing Integrity, Privacy can be added later).
</Note>

## TL;DR

- **Ready today:** core encryption, IRSA-based keyless auth, protected-branch CI/CD, subprocessor list, in-app TOTP for end-users, Loki + Prometheus observability.
- **Blocks Type 1:** no CloudTrail, no IAM MFA for team members, no password policy, no GuardDuty/Security Hub/Config, no vulnerability scanning in CI, no WAF, no written policies, no formal incident response.
- **Estimated cost:** $15k–30k/yr compliance-automation platform (Vanta/Drata/Secureframe), $10k–25k for auditor (Type 1 + Type 2), plus small AWS service uplift (~$100/mo).
- **Effort:** ~1 FTE for 10–14 weeks to get to Type 1. Thereafter continuous evidence collection.

## What is SOC 2 (30-second version)

SOC 2 is an **auditor's attestation** that your controls work as advertised for at least one of five Trust Service Criteria:

1. **Security** — required. Protection against unauthorized access.
2. **Availability** — uptime commitments.
3. **Confidentiality** — protection of customer-designated confidential data.
4. **Processing Integrity** — operations are complete, valid, accurate.
5. **Privacy** — personal information is handled per the privacy notice.

**Type 1** = controls are designed correctly at a point in time. **Type 2** = controls operated effectively over 3–12 months. Most enterprise buyers accept Type 1 short-term and require Type 2 after a year.

## Phased roadmap

### Phase 0: AWS-level fixes (Week 1) {#phase-0}

One focused half-day. Cheap, high-value, no code changes.

<Steps>
  <Step title="Enable CloudTrail (multi-region, log file validation on)">
    ```bash
    aws cloudtrail create-trail \
      --name skygen-management-events \
      --s3-bucket-name skygen-cloudtrail-logs-us \
      --is-multi-region-trail \
      --enable-log-file-validation \
      --region us-east-1
    aws cloudtrail start-logging --name skygen-management-events --region us-east-1
    ```
    Create the log bucket with object-lock (WORM) if possible. CloudTrail costs: first trail free for management events, ~$2/100k events for data events.
  </Step>
  <Step title="Enable IAM password policy">
    ```bash
    aws iam update-account-password-policy \
      --minimum-password-length 14 \
      --require-symbols \
      --require-numbers \
      --require-uppercase-characters \
      --require-lowercase-characters \
      --max-password-age 90 \
      --password-reuse-prevention 10 \
      --allow-users-to-change-password
    ```
  </Step>
  <Step title="Enforce IAM MFA for all human users">
    For each of the 9 users: walk them through adding a virtual MFA device (Google Authenticator / 1Password / Authy).
    ```bash
    # Attach deny-without-mfa policy to all human users
    aws iam put-user-policy --user-name Arseniy \
      --policy-name DenyAllExceptListedIfNoMFA \
      --policy-document file://require-mfa-policy.json
    ```
    Standard AWS MFA-enforcement policy: [docs.aws.amazon.com/IAM/latest/UserGuide/tutorial_users-self-manage-mfa-and-creds.html](https://docs.aws.amazon.com/IAM/latest/UserGuide/tutorial_users-self-manage-mfa-and-creds.html)
  </Step>
  <Step title="Enable GuardDuty, Security Hub, Access Analyzer, Config">
    ```bash
    # GuardDuty
    aws guardduty create-detector --enable --region us-east-1

    # Security Hub (auto-enables AWS Foundational + CIS benchmarks)
    aws securityhub enable-security-hub --enable-default-standards --region us-east-1

    # IAM Access Analyzer (account-level)
    aws accessanalyzer create-analyzer --analyzer-name skygen-account --type ACCOUNT --region us-east-1

    # Config (record all supported resources)
    aws configservice put-configuration-recorder \
      --configuration-recorder name=default,roleARN=arn:aws:iam::118183257315:role/aws-service-role/config.amazonaws.com/AWSServiceRoleForConfig,recordingGroup.allSupported=true \
      --region us-east-1
    aws configservice start-configuration-recorder --configuration-recorder-name default --region us-east-1
    ```
    Combined add-on cost: ~$30–80/mo depending on resource volume.
  </Step>
  <Step title="Turn on RDS DeletionProtection and S3 Versioning on critical buckets">
    ```bash
    for db in skygen-prod-db skygen-dev-db1-use1-eks database-landing-prod; do
      aws rds modify-db-instance --db-instance-identifier $db --deletion-protection --apply-immediately --region us-east-1
    done
    for b in skygen-files-prod skygen-screenshots-us skygen-cloudpc-profiles-prod; do
      aws s3api put-bucket-versioning --bucket $b --versioning-configuration Status=Enabled
    done
    ```
  </Step>
  <Step title="Encrypt `database-landing-dev`">
    No in-place encryption on RDS — take encrypted snapshot, restore as new instance, cut over.
  </Step>
</Steps>

Cost: ~$80–150/mo ongoing AWS uplift. Effort: 1 engineer, 1 day.

### Phase 1: Code & CI/CD hardening (Weeks 2–3) {#phase-1}

<Steps>
  <Step title="Add GitHub Dependabot config">
    In every repo root — `.github/dependabot.yml`:
    ```yaml
    version: 2
    updates:
      - package-ecosystem: pip
        directory: "/Backend"
        schedule: { interval: weekly }
      - package-ecosystem: pip
        directory: "/Agent"
        schedule: { interval: weekly }
      - package-ecosystem: npm
        directory: "/WebApp"
        schedule: { interval: weekly }
      - package-ecosystem: docker
        directory: "/Backend"
        schedule: { interval: weekly }
    ```
  </Step>
  <Step title="Enable CodeQL (static analysis)">
    Add `.github/workflows/codeql.yml` using GitHub's default CodeQL setup. Covers Python + JavaScript/TypeScript for free on public repos, Advanced Security on private.
  </Step>
  <Step title="Add secret scanning">
    GitHub Advanced Security → Secret scanning on. If not licensed: use Gitleaks as a CI step.
  </Step>
  <Step title="Add container vulnerability scanning in CI">
    After `docker push`, run Trivy or AWS ECR's built-in scan-on-push:
    ```bash
    aws ecr put-image-scanning-configuration \
      --repository-name skygen/backend \
      --image-scanning-configuration scanOnPush=true \
      --region us-east-1
    # repeat for all 7 repos
    ```
  </Step>
  <Step title="Publish SECURITY.md">
    Root-level `SECURITY.md` with vulnerability-disclosure address (e.g., `security@skygen.ai`), response SLA, and scope.
  </Step>
  <Step title="Tighten branch protection">
    Required on `main` / `develop`:
    - Pull request required, 1+ approver
    - Status checks must pass (CodeQL, tests, Dependabot alerts reviewed)
    - Dismiss stale approvals on new commits
    - Require signed commits (optional, recommended)
    - No force-push
  </Step>
</Steps>

Effort: 1 engineer, 3–5 days.

### Phase 2: Documented policies (Weeks 3–6) {#phase-2}

SOC 2 requires *written, approved, communicated* policies. Draft from templates, have the CEO/CTO sign, store in `mintlify-docs/security/policies/`.

Minimum policy set:

1. **Information Security Policy** — top-level commitment
2. **Access Control Policy** — who gets what, review cadence
3. **Data Classification Policy** — what's confidential, how it's labeled
4. **Acceptable Use Policy** — what employees can and can't do on company systems
5. **Incident Response Plan** — detection, triage, escalation, RCA, post-mortem
6. **Business Continuity / Disaster Recovery Plan** — RTO, RPO, test schedule
7. **Change Management Policy** — the PR + review + deploy process you already use, written down
8. **Vendor Management Policy** — how subprocessors are added (the [Subprocessors](/security/subprocessors) page plus a vetting checklist)
9. **Data Retention & Destruction Policy** — how long we keep what, how we delete
10. **Secure SDLC Policy** — coding standards, review requirements, dependency hygiene
11. **Risk Assessment Policy** — annual risk register + review
12. **HR / Personnel Security Policy** — onboarding checklist, background-check requirement, training, offboarding
13. **Physical Security Policy** — laptops, office access (even remote-first teams need this)
14. **Cryptographic Controls Policy** — what's encrypted, how keys are managed
15. **Privacy / Data Subject Rights Policy** — GDPR/CCPA handling

Don't write from scratch. Start from:
- Compliance-automation platforms (Vanta/Drata/Secureframe) bundle ~40 template policies.
- **Free alternative:** [JupiterOne policy templates](https://github.com/JupiterOne/security-policy-templates) (MIT-licensed).

Effort: 20–40 hours over 3 weeks, split across founder-level approvers.

### Phase 3: Compliance-automation platform (Week 4 onwards, parallel) {#phase-3}

An automation platform collects evidence continuously so Type 2 is not a scramble at the end of the observation window. Three leaders in the space:

| Platform | Strength | Indicative price (50 employees) |
|----------|----------|------------------------------|
| **Vanta** | Most mature, biggest integration library, SOC 2 playbooks are opinionated | ~$11k/yr base + SOC 2 add-on |
| **Drata** | Strongest automation of evidence collection, good UX | ~$14k/yr |
| **Secureframe** | Cheaper entry point, solid for SOC 2 + ISO 27001 dual track | ~$7k/yr base |
| **Sprinto** | Startup-focused, best budget option | ~$6k/yr |

All four integrate with AWS (via read-only IAM role), GitHub, Google Workspace, Slack, MDM, Okta, etc. They auto-import users, run daily checks, produce auditor-ready evidence.

**Recommendation:** pick **Vanta** if you can afford it, **Secureframe** or **Sprinto** if budget-constrained. All three will cut audit prep time by ~60%.

### Phase 4: People & process (Weeks 6–10) {#phase-4}

- **Background checks** on new hires (e.g., Certn, Checkr) — $50–100 per hire.
- **Annual security training** — KnowBe4, Ninjio, or free Federal Virtual Training Environment + attestation form.
- **Phishing simulation** — KnowBe4 or Hoxhunt.
- **MDM for company laptops** — Kandji (macOS) or Fleet/Jamf. Enforces disk encryption, screen lock, OS updates.
- **Quarterly access reviews** — each manager confirms their team's AWS / GitHub / SaaS permissions.
- **Quarterly IR tabletop exercise** — walk through a ransomware or data breach scenario, record outcomes.
- **Onboarding / offboarding runbooks** — checklist with SaaS accounts, SSH keys, AWS IAM, GitHub.

### Phase 5: Penetration test (Week 10–12) {#phase-5}

Most customers will ask for a third-party pentest report at SOC 2 time. Vendors:

- **Cobalt** — Pentest-as-a-service, ~$8k–15k per engagement, 1–2 weeks turnaround
- **Bishop Fox** — top-tier, $25k+, 2–3 weeks
- **HackerOne / Bugcrowd** — bug-bounty model, pricing variable

Plan to remediate findings before auditor comes on site.

### Phase 6: Select auditor & run Type 1 (Weeks 12–18) {#phase-6}

**Auditor shortlist** — firms commonly used by YC-stage SaaS:

- **Prescient Assurance** — fast, startup-friendly, Vanta/Drata integrated
- **A-LIGN** — large, well-known, slower
- **Johanson Group** — mid-size, good pricing
- **BARR Advisory** — tech-focused, deep cloud expertise

Typical Type 1 price: $10k–18k. Type 2 (first year): $15k–25k. Observation window: 3 months minimum, 6 months more commonly accepted.
