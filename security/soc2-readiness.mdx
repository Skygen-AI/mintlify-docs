## Overview

Skygen is working toward SOC 2 Type 1 compliance. This document tracks the current state of security controls across the five Trust Service Criteria (Security, Availability, Processing Integrity, Confidentiality, Confidentiality). It identifies existing controls, known gaps, and remediation priorities.

## Remediation roadmap

### Phase 1 — Type 1 blockers (30 days)

1. Enable MFA for all AWS IAM users
2. Set IAM account password policy (min 12 chars, require uppercase, lowercase, numbers, symbols, 90-day rotation)
3. Enable CloudTrail in all regions, ship to S3
4. Enable AWS WAF on the ALB with OWASP managed rule group
5. Enable GuardDuty on the AWS account
6. Rotate all stale IAM access keys

### Phase 2 — Operational controls (60 days)

7. Enable ECR image scanning on push
8. Add automated dependency scanning (e.g., `pip-audit`, `npm audit`) to CI pipeline
9. Document data retention policy and implement automated purge for deleted sessions
10. Write formal incident response plan
11. Enable AWS Config with CIS AWS Foundations benchmark rules

### Phase 3 — Type 2 preparation (90 days)

12. Engage pen testing firm for external assessment
13. Implement formal on-call rotation with runbooks
14. Enable S3 versioning on all buckets
15. Implement mTLS for internal pod-to-pod communication

## Phase 1 remediation commands

The following AWS CLI commands address the Type 1 blockers listed above. Run these from a shell with administrator-level AWS credentials.

### Enable CloudTrail

```bash
# Create S3 bucket for CloudTrail logs
aws s3api create-bucket \
  --bucket skygen-cloudtrail-logs \
  --region us-east-1

# Apply bucket policy to allow CloudTrail writes
aws s3api put-bucket-policy \
  --bucket skygen-cloudtrail-logs \
  --policy file://cloudtrail-bucket-policy.json

# Create multi-region trail
aws cloudtrail create-trail \
  --name skygen-audit \
  --s3-bucket-name skygen-cloudtrail-logs \
  --is-multi-region-trail \
  --include-global-service-events \
  --enable-log-file-validation

# Start logging
aws cloudtrail start-logging --name skygen-audit
```

### Enforce IAM password policy

```bash
aws iam update-account-password-policy \
  --minimum-password-length 14 \
  --require-symbols \
  --require-numbers \
  --require-uppercase-characters \
  --require-lowercase-characters \
  --allow-users-to-change-password \
  --max-password-age 90 \
  --password-reuse-prevention 12 \
  --hard-expiry
```

### Enable GuardDuty

```bash
# Enable GuardDuty detector in us-east-1
aws guardduty create-detector \
  --enable \
  --finding-publishing-frequency FIFTEEN_MINUTES \
  --region us-east-1

# Retrieve detector ID
DETECTOR_ID=$(aws guardduty list-detectors --region us-east-1 \
  --query 'DetectorIds[0]' --output text)
echo "GuardDuty detector: $DETECTOR_ID"
```

### Enable WAF on ALB

```bash
# Create WAF Web ACL with OWASP managed rule group
aws wafv2 create-web-acl \
  --name skygen-waf \
  --scope REGIONAL \
  --region us-east-1 \
  --default-action Allow={} \
  --rules '[
    {
      "Name": "AWSManagedRulesCommonRuleSet",
      "Priority": 1,
      "OverrideAction": {"None": {}},
      "Statement": {
        "ManagedRuleGroupStatement": {
          "VendorName": "AWS",
          "Name": "AWSManagedRulesCommonRuleSet"
        }
      },
      "VisibilityConfig": {
        "SampledRequestsEnabled": true,
        "CloudWatchMetricsEnabled": true,
        "MetricName": "AWSManagedRulesCommonRuleSetMetric"
      }
    }
  ]' \
  --visibility-config SampledRequestsEnabled=true,CloudWatchMetricsEnabled=true,MetricName=skygen-waf

# Associate with the ALB
ALB_ARN=$(aws elbv2 describe-load-balancers \
  --query 'LoadBalancers[?contains(LoadBalancerName, `skygen`)].LoadBalancerArn' \
  --output text)

WAF_ACL_ARN=$(aws wafv2 list-web-acls --scope REGIONAL --region us-east-1 \
  --query 'WebACLs[?Name==`skygen-waf`].ARN' --output text)

aws wafv2 associate-web-acl \
  --web-acl-arn $WAF_ACL_ARN \
  --resource-arn $ALB_ARN \
  --region us-east-1
```

## Existing compliant controls

The following controls are already in place and can serve as evidence in an audit engagement.

### Evidence for CC6 (Logical and Physical Access)

- **JWT authentication**: All API requests require `Authorization: Bearer <token>`. Tokens are signed with `HS256` using `SECRET_KEY` from AWS Secrets Manager. Implementation: `Backend/app/security.py`.
- **IRSA for service accounts**: Pods access AWS services (S3, Secrets Manager) through IAM Roles for Service Accounts, not long-lived key pairs. IAM role annotations in `Backend/k8s/backend/deployment.yaml`.
- **Per-user data isolation**: Every database query filters by `user_id`. No cross-user data access is possible at the API layer. Evidence: repository layer in `Backend/app/repositories/`.

### Evidence for CC7 (System Operations)

- **Application logging**: All HTTP requests logged with caller type, latency, and status code via `logging_middleware.py`. Logs shipped to Loki (`skygen-us-dev` and `skygen-us-prod` clusters).
- **Alerting pipeline**: Prometheus → Alertmanager → Telegram for all `severity: critical` alerts. Alertmanager config in `Backend/k8s/monitoring/`.
- **Safety violation audit trail**: All `UNSAFE` and `NEED_HUMAN_CLARIFICATION` verdicts persisted to the `safety_violations` table with user, task, input, and output captured.

### Evidence for CC9 (Risk Mitigation)

- **Connector secret encryption**: All custom connector credentials encrypted at rest using AES via `MCP_CRYPTO_KEY`. API responses redact secrets as `[REDACTED]`. Implementation: `Backend/app/routers/connectors_crypto.py`.
- **SSRF protection**: All `base_url` and `spec_url` values in connector creation are validated against private IP ranges (RFC 1918, link-local, loopback) before saving. Implementation: `validate_url_not_ssrf()`.
- **Webhook HMAC verification**: All inbound Composio webhooks are verified using HMAC-SHA256 with `COMPOSIO_WEBHOOK_SECRET` before processing.
